# 02_Pointcut_기반_AOP_적용_여부_검토

## 상위 맥락
03_내부_동작_흐름 < 03_AOP_작성_및_동작_원리 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
**'Pointcut 기반 AOP 적용 여부 검토'**는 스프링이 애플리케이션 초기화 단계에서 타겟(Target) 객체에 부가 기능(Advice)을 엮기 위해 프록시(Proxy)를 생성할지 여부를 결정하는 핵심적인 내부 동작 흐름입니다.

스프링 AOP의 전체적인 내부 동작 원리와 인과관계 속에서 이 검토 단계가 갖는 역할과 흐름은 다음과 같이 전개됩니다.
### 1. 빈(Bean) 설정 분석 및 생성 스프링 컨테이너는 애플리케이션 구동 시 등록된 설정들을 분석하여 실제 사용할 빈(Bean) 객체들을 메모리에 생성합니다.
이 시점에 생성된 객체는 아직 어떠한 부가 기능도 엮이지 않은, 순수한 비즈니스 로직만을 가진 타겟(Target) 원본 객체입니다.
### 2. Pointcut 기반 AOP 적용 검토 (필터링 평가) 빈이 생성된 직후, 컨테이너는 시스템에 등록된 관점(Aspect)들의 포인트컷(Pointcut) 표현식을 필터로 사용하여 방금 생성된 빈이 부가 기능을 적용받아야 하는 대상인지 정밀하게 검토합니다.
즉, 이 빈의 클래스 경로나 메서드 시그니처 등이 execution과 같은 포인트컷 조건에 부합하는 조인 포인트(Join point)를 가지고 있는지 일일이 대조하여 확인하는 과정입니다.

### 3. 프록시(Proxy) 생성 및 노출 대상 변경 검토 결과, 해당 빈이 하나라도 포인트컷 조건에 부합하여 AOP 적용 대상임이 확인되면, 스프링은 원본 타겟 객체 대신 이를 감싸는 대리 객체인 프록시(Proxy)를 동적으로 생성합니다.
이후 내부 동작 흐름에서 가장 결정적인 **'노출 대상 변경(target -> proxy)'**이 일어납니다.
스프링 컨테이너는 의존성 주입 시 클라이언트에게 진짜 타겟 객체가 아닌 이 프록시 객체를 빈으로 주입하여 노출시킵니다.

### 4. 런타임 시 내부적인 동작 (호출 가로채기 및 실행) 이러한 초기 검토와 대체 흐름의 결과로, 런타임에 클라이언트가 타겟의 메서드(예: mDao.selectDetail)를 호출하게 되면, 그 호출은 원본 타겟이 아닌 프록시 객체에게 도달하여 가로채어집니다.
프록시는 지정된 타이밍에 부가 기능(예: LoggingAspect의 @Before 로그 출력)을 먼저 처리한 후, 비로소 자신이 감싸고 있던 실제 타겟(예: MemberDao)의 비즈니스 로직을 위임하여 실행시킵니다.

정리하면 **'Pointcut 기반 AOP 적용 여부 검토'**는 순수한 객체를 AOP의 동적 위빙(Weaving) 생태계로 편입시키기 위한 초기화 시점의 판별 및 변환 분기점입니다.
스프링 컨테이너가 배후에서 포인트컷을 기반으로 대상을 엄격히 검토하고 런타임 노출 대상을 프록시로 완전히 바꿔치기하기 때문에, 클라이언트와 타겟 객체 양측은 AOP의 개입 여부를 전혀 의식하지 않고도 완벽하게 모듈화된 부가 기능을 실행할 수 있게 됩니다.
