# 01_빈_설정_분석_및_빈_생성

## 상위 맥락
03_내부_동작_흐름 < 03_AOP_작성_및_동작_원리 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(Spring AOP)의 아키텍처에서 '빈 설정 분석'과 '빈 생성'은 핵심 비즈니스 로직(Target)의 소스코드를 전혀 수정하지 않고도 부가 기능(Aspect)을 런타임에 동적으로 결합(Weaving)해 내기 위한 내부 동작 흐름의 가장 기초적이고 필수적인 첫 단계입니다.
AOP의 Proxy 동작 원리와 내부 동작 흐름은 프록시 객체의 생성과 활용으로 이어지는 구조를 가집니다.
### 1. 빈 설정 분석 (Bean Configuration Analysis) 스프링 컨테이너(IoC Container)가 애플리케이션을 구동할 때, 가장 먼저 수행하는 작업은 객체들에 대한 설정 정보를 읽어 들이는 '빈 설정 분석'입니다.
이 단계에서 스프링은 @Component, @Service 등이 선언된 일반적인 비즈니스 로직(잠재적인 Target)뿐만 아니라, @Aspect가 선언된 횡단 관심사 모듈(어드바이스와 포인트컷을 포함)까지 모두 스캔하여 컨테이너가 관리해야 할 대상 객체들의 구조를 파악합니다.

### 2. 빈 생성 (Bean Creation) 설정 분석이 완료되면, 스프링은 해당 명세에 따라 메모리상에 실제 객체(빈)들을 인스턴스화하는 '빈 생성' 과정을 거칩니다.
이 시점에서 생성된 순수한 비즈니스 빈은 아직 어떠한 부가 기능과도 결합되지 않은 원래의 코드 상태(Target)를 유지하고 있습니다.
### 3. Pointcut 기반 AOP 적용 검토 (대상 필터링) 빈이 생성된 직후, 스프링은 단순한 의존성 주입으로 넘어가지 않고 AOP 아키텍처를 위한 결정적인 중간 개입을 수행합니다. 생성된 모든 빈들을 대상으로, 사전에 분석해 둔 Aspect 내부의 Pointcut 표현식 조건에 부합하는지(즉, 부가 기능을 적용해야 할 대상인지)를 엄격하게 검토합니다.

### 4. Proxy 생성 및 노출 대상 변경 (동적 위빙의 핵심) 검토 결과, 특정 생성된 빈(Target)이 Pointcut의 조건에 해당하여 Advice(부가 기능)를 적용해야 한다고 판단되면, 스프링은 원본 Target 객체를 그대로 컨테이너에 등록하지 않습니다. 대신 원본 Target을 감싸고 제어할 수 있는 가짜 대리 객체인 '프록시(Proxy)'를 동적으로 생성합니다.
이후 스프링 컨테이너는 클라이언트에게 제공할 **'노출 대상'을 원래의 Target에서 방금 생성한 Proxy로 완벽하게 변경(치환)**해 버립니다.

### 5. 런타임 시점의 내부적인 동작 흐름 (ctx.getBean()의 반환 및 호출)
이러한 일련의 빈 생성 및 대체 흐름 결과, 클라이언트가 ctx.getBean()을 통해 컨테이너에 특정 빈을 요청하면, 스프링은 실제 순수 비즈니스 객체(Target)가 아닌 위장된 Proxy 객체를 반환하게 됩니다.

따라서 클라이언트가 비즈니스 메서드(예: mDao.selectDetail)를 호출하더라도, 이 호출은 프록시 객체에 의해 먼저 가로채어집니다(Intercept).
프록시는 가로챈 호출 흐름 속에서 지정된 타이밍(예: @Before)에 Advice 로직(예: 로깅)을 수행한 후, 내부적으로 진짜 Target 객체의 메서드를 위임받아 실행하는 방식으로 완벽한 관심사의 결합을 이루어냅니다.

정리하면 스프링 AOP의 내부 동작 흐름에서 '빈 설정 분석 및 빈 생성' 단계는 단순한 객체의 초기화 과정이 아닙니다.
이는 스프링 컨테이너가 런타임에 빈의 생명주기(Lifecycle)에 적극적으로 개입하여, 원본 비즈니스 코드를 훼손하지 않으면서도 프록시(Proxy) 기반의 횡단 관심사를 엮어낼(Weaving) 수 있도록 판을 까는 핵심적인 준비 단계입니다.
이 완벽한 제어의 역전(IoC) 덕분에 개발자는 부가 기능의 동작 원리에 신경 쓰지 않고 비즈니스 로직 작성에만 집중할 수 있는 진정한 의미의 관점 지향 프로그래밍이 완성됩니다.
