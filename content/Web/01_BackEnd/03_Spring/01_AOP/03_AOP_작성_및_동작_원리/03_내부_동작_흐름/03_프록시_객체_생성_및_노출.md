# 03_프록시_객체_생성_및_노출

## 상위 맥락
03_내부_동작_흐름 < 03_AOP_작성_및_동작_원리 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(관점 지향 프로그래밍)의 내부 동작 흐름에서 '프록시(Proxy) 객체의 생성 및 노출'은 횡단 관심사(Aspect)를 핵심 비즈니스 로직(Target)에 엮어내는(Weaving) 가장 핵심적인 물리적 메커니즘입니다.
이 메커니즘이 전체 내부 동작 흐름 속에서 어떻게 전개되는지 단계별 인과관계로 살펴보면 다음과 같습니다.
### 1. 빈(Bean) 설정 분석 및 AOP 적용 검토 스프링 컨테이너가 애플리케이션을 구동하며 빈을 생성하는 초기화 단계에서, 시스템은 등록된 빈들의 설정을 분석합니다.
이때 개발자가 작성해 둔 포인트컷(Pointcut) 조건을 평가하여, 현재 생성 중인 빈이 AOP 부가 기능(Advice)을 적용해야 할 대상(Target)인지 검토합니다.

### 2. 동적 프록시(Proxy) 객체 생성 검토 결과 해당 빈이 AOP 적용 대상이라면, 스프링은 순수한 비즈니스 로직을 가진 원본 타겟 객체 외에 타겟을 감싸는 대리인 객체인 '프록시(Proxy)'를 런타임에 동적으로 생성합니다.
기본적으로 스프링은 CGLib(Code Generator Library)를 활용하여 타겟 클래스를 상속받는 방식의 클래스 기반 프록시를 생성하며, 필요에 따라 인터페이스 기반의 JdkDynamicProxy 방식을 사용할 수도 있습니다.

### 3. 노출 대상의 변경 (Target -> Proxy) 이 단계가 내부 동작 흐름의 구조적 전환점입니다. 스프링 컨테이너는 클라이언트에게 원본 타겟(Target) 빈 대신, 방금 생성한 프록시(Proxy) 빈을 반환하여 노출 대상을 완전히 변경합니다.
즉, 클라이언트가 ctx.getBean() 등을 통해 객체를 요청할 때, 클라이언트 입장에서는 일반적인 요청인지 프록시를 통한 요청인지 전혀 알 수 없는 투명한 상태가 유지됩니다.
실제로 런타임에 주입된 빈의 클래스명을 확인해 보면 원본 클래스 이름 뒤에 $$SpringCGLIB$$0과 같은 꼬리표가 붙은 프록시 객체가 적용되어 노출되고 있음을 확인할 수 있습니다.

### 4. 런타임 내부 동작: 요청 가로채기와 위임 노출 대상이 프록시로 변경되었기 때문에, 클라이언트가 비즈니스 메서드(예: mDao.selectDetail)를 호출하면 원본 타겟이 아닌 프록시가 가장 먼저 이 호출을 가로챕니다(Intercept).
프록시는 내부에 정의된 어드바이스 로직(예: LoggingAspect의 로깅 기능)을 지정된 시점(예: @Before)에 먼저 수행한 뒤, 내부적으로 참조하고 있는 실제 타겟 객체의 메서드(예: return target.insert(...))를 직접 호출하여 비즈니스 로직 처리를 위임합니다.

정리 결과적으로 내부 동작 흐름 상에서 '프록시 객체의 생성과 노출 대상 변경'은 원본 비즈니스 코드를 단 한 줄도 훼손하지 않으면서도, 런타임 환경에서 애플리케이션의 제어 흐름을 AOP 인프라가 완전히 장악하도록 만드는 결정적 전략입니다.
프록시가 클라이언트와 타겟 사이에서 완벽한 대리자 역할을 수행함으로써, 비즈니스 로직은 철저하게 고립되어 순수성을 유지하고 시스템 전반에는 모듈화된 부가 기능이 매끄럽게 엮이게(Weaving) 됩니다.
