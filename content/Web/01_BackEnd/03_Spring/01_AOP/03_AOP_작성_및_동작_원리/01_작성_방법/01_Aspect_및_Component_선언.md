# 01_Aspect_및_Component_선언

## 상위 맥락
01_작성_방법 < 03_AOP_작성_및_동작_원리 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 환경에서 AOP(관점 지향 프로그래밍)를 실제 코드로 구현하는 **작성 방법(Implementation)**의 가장 기본적이고 핵심적인 단계는 횡단 관심사를 모듈화할 클래스 상단에 @Aspect와 @Component 애너테이션을 함께 선언하는 것입니다.

이 두 애너테이션은 단순한 선언을 넘어, 각각 AOP 생태계와 스프링 프레임워크 컨테이너 관점에서 명확하고 독립적인 역할을 수행합니다.
이 두 선언이 갖는 구체적인 의미와 시스템 동작의 인과관계는 다음과 같습니다.
### 1. @Aspect: 횡단 관심사 모듈로서의 정체성 부여 클래스 레벨에 @Aspect를 선언하는 것은 해당 객체가 단순한 비즈니스 로직(Target)용 객체가 아니라, 하나 이상의 어드바이스(Advice) 로직을 묶어놓은 횡단 관심사의 논리적 모듈(Aspect)임을 명시하는 것입니다.
개발자는 이 선언이 붙은 클래스 내부에 @Before, @AfterReturning 등의 어드바이스 메서드를 작성하여 '무엇을 언제' 동작할지 정의하고, 이를 적용할 대상을 필터링하는 포인트컷(Pointcut) 표현식을 함께 구성합니다.
그러나 @Aspect라는 선언 자체는 AOP의 문법적 역할만을 규정할 뿐, 스프링 프레임워크가 이 객체를 실제로 메모리에 올려 관리하게 만들지는 못합니다.
### 2. @Component: 스프링 빈(Bean) 등록을 통한 생명주기 및 위빙 제어권 확보 스프링 AOP 메커니즘의 가장 큰 기술적 특징은 타겟(Target) 객체의 실제 코드를 직접 조작하지 않고, 런타임 시점에 횡단 관심사가 결합된 대리 객체인 프록시(Proxy) 빈을 생성하여 간접적으로 동작한다는 것입니다.
따라서 작성된 Aspect 모듈이 타겟 객체의 조인 포인트(Join point)에 동적으로 엮이기(Weaving) 위해서는, 반드시 해당 Aspect 클래스 자체도 스프링 컨테이너가 생명주기를 직접 관리하는 '하나의 빈(Bean)'으로 등록되어야만 합니다.
이를 위해 @Component 애너테이션을 추가로 선언합니다.
작성 방법의 전체적인 아키텍처 흐름 및 인과관계 이 두 가지 애너테이션 선언을 기점으로 AOP는 다음과 같은 논리적 인과관계를 거쳐 동작합니다.
모듈 작성: 개발자는 로깅, 보안 등의 부가 기능을 수행할 클래스를 만들고 @Component와 @Aspect를 부여합니다.

컨테이너 초기화 및 인식: 애플리케이션 구동 시 스프링 컨테이너는 컴포넌트 스캔을 통해 이 클래스를 빈으로 등록함(@Component)과 동시에, 이 빈이 일반 빈이 아닌 AOP 관점 객체(@Aspect)임을 인식합니다.

프록시 생성 및 런타임 결합: 스프링은 식별된 Aspect 내부에 작성된 포인트컷 조건을 분석하여, 부가 기능이 적용되어야 할 타겟 빈들을 확인합니다.
이후 클라이언트 호출을 대신 받을 프록시(Proxy) 객체를 자동 생성하며, 런타임에 프록시가 타겟 메서드의 실행을 가로채어 Aspect 내부의 어드바이스 로직을 실행시킵니다.

정리 결과적으로 AOP 작성 방법론에서 @Aspect는 클래스의 구조적 목적(횡단 관심사의 캡슐화)을 정의하고, @Component는 스프링의 프록시 기반 메커니즘이 해당 관점을 인식하여 런타임에 조작할 수 있도록 시스템적 기반을 마련해 줍니다.
이 두 선언이 결합됨으로써, 개발자가 텍스트로 작성한 부가 기능 코드가 핵심 비즈니스 로직을 전혀 침범하지 않으면서도 강력하고 유연하게 제어권을 행사하는 AOP 모듈로 작동하게 됩니다.
