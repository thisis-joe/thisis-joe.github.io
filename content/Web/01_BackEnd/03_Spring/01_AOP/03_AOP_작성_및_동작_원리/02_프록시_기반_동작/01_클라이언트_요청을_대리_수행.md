# 01_클라이언트_요청을_대리_수행

## 상위 맥락
02_프록시_기반_동작 < 03_AOP_작성_및_동작_원리 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(관점 지향 프로그래밍) 아키텍처에서 '클라이언트 요청을 대리 수행'한다는 것은 프록시(Proxy) 기반 동작 원리의 본질을 가장 명확하게 설명하는 개념입니다.
프록시(Proxy)의 사전적 의미는 '특정 사람이나 조직을 대신해서 행동하는 대리인'이며, 시스템 네트워크 상에서는 **'클라이언트를 대신해서 요청을 수행하는 서버(또는 객체)'**를 의미합니다.
이러한 대리 수행 메커니즘이 Spring AOP의 프록시 기반 동작이라는 더 큰 맥락에서 어떻게 구현되고 어떤 구조적 가치를 지니는지 구체적으로 살펴보면 다음과 같습니다.
### 1. 원본 객체(Target)의 은폐와 대리 객체(Proxy)의 반환 Spring AOP는 부가 기능을 원본 코드에 직접 삽입하지 않고, 프록시(Proxy) 기반의 빈(Bean)을 생성하여 AOP를 구현합니다.
애플리케이션이 구동될 때 스프링 컨테이너는 빈 설정과 Pointcut 조건들을 분석합니다.
만약 특정 빈이 AOP 적용 대상(Target)으로 판별되면, 스프링은 클라이언트(ctx.getBean())에게 순수한 비즈니스 로직을 가진 원본 Target 객체를 직접 내어주지 않습니다.
대신 Target을 감싸고 있는 '대리인'인 Proxy 객체를 생성하여 반환합니다.

### 2. 요청의 가로채기와 위임(Delegation) 프로세스 클라이언트가 Target의 특정 메서드(예: selectDetail())를 호출했다고 가정해 보겠습니다.
가로채기(Intercept): 클라이언트는 자신이 실제 Target과 통신한다고 생각하지만, 실제로는 프록시가 클라이언트의 요청을 중간에서 가로채어 대신(대리) 접수합니다.

부가 기능(Advice) 수행: 요청을 가로챈 프록시는 자신이 감싸고 있는 Target으로 흐름을 넘기기 전이나 후에, 설정된 횡단 관심사 로직(로깅, 트랜잭션 등)을 실행합니다.

실제 대상에게 위임: 부가 기능 수행이 끝나면, 프록시는 내부적으로 진짜 비즈니스 로직을 보유한 Target의 원본 메서드를 호출(target.insert() 등)하여 본래 클라이언트가 의도했던 핵심 업무를 위임받아 처리하게 합니다.

### 3. 완전한 투명성(Transparency) 제공 이러한 프록시의 '클라이언트 요청 대리 수행'이 갖는 가장 강력한 기술적 특징은 투명성입니다. 클라이언트나 업무를 요청하는 입장에서는 자신이 일반적인 원본 객체에게 요청을 보낸 것인지, 아니면 Proxy를 거쳐 요청이 처리되고 있는지 전혀 알 수 없습니다(알 필요도 없습니다).
클라이언트 코드는 아무런 변경 없이 기존 인터페이스를 그대로 사용할 수 있습니다.
정리 결과적으로 Spring AOP에서 **"클라이언트의 요청을 대리 수행한다"**는 것은 부가 기능과 핵심 기능의 물리적 결합을 런타임에 끊어내는 핵심 전략입니다.
프록시가 중간에 개입하여 클라이언트의 모든 호출을 대리하고 흐름을 제어해 주기 때문에, 정작 시스템의 코어인 Target 객체는 주변의 부가 기능이나 자신을 호출하는 클라이언트를 전혀 신경 쓰지 않고 오직 순수 비즈니스 로직에만 몰입할 수 있습니다.
즉, 프록시의 완벽한 대리 수행 역할 덕분에 AOP가 지향하는 극단적인 수준의 '관심사의 분리(Separation of Concerns)'가 실현되는 것입니다.
