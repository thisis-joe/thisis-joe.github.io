# 02_JDK_Dynamic_Proxy_인터페이스_기반

## 상위 맥락
02_프록시_기반_동작 < 03_AOP_작성_및_동작_원리 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(Spring AOP)는 핵심 비즈니스 로직(Target)의 코드를 직접 수정하지 않고, 그 역할을 대신 수행하는 대리 객체인 프록시(Proxy) 기반의 빈(Bean)을 이용하여 런타임에 부가 기능을 구현(Weaving)하는 방식으로 동작합니다.
클라이언트가 빈을 요청하면 스프링 컨테이너는 원래의 Target 객체 대신 Proxy 객체를 반환하도록 노출 대상을 변경합니다.
따라서 업무를 처리하는 클라이언트 입장에서는 자신이 호출하는 대상이 순수한 일반 객체(Target)인지 프록시(Proxy)를 통한 것인지 전혀 알 수 없는 상태로 요청을 수행하게 됩니다.

이러한 프록시 기반의 동작 원리 속에서 JDK Dynamic Proxy(인터페이스 기반)는 스프링이 런타임에 가짜 대리 객체(Proxy)를 만들어내는 핵심 기술 전략 중 하나로 자리 잡고 있습니다.
구체적인 동작 방식과 원리는 다음과 같습니다.
### 1. 프록시 생성 전략의 분기: CGLib vs JDK Dynamic Proxy 스프링 AOP가 프록시 객체를 생성할 때 기본적으로 취하는 동작은 대상 클래스 자체를 상속받아 프록시를 만드는 CGLib(Code Generator Library) 기반의 클래스(Class) 상속 방식입니다.
하지만, 타겟(Target) 객체가 특정 인터페이스(Interface)를 구현하고 있는 구조라면, 스프링은 해당 인터페이스를 기반으로 프록시를 동적으로 생성하는 JDK Dynamic Proxy 방식을 사용할 수 있습니다.

### 2. 인터페이스 기반(Interface-based) 동작의 의미 JDK Dynamic Proxy가 '인터페이스 기반'이라는 것은, 생성된 프록시 객체가 실제 타겟 객체의 클래스 타입이 아닌 그 타겟이 구현하고 있는 인터페이스 타입을 기반으로 만들어진다는 것을 의미합니다. 클라이언트는 인터페이스를 통해 메서드를 호출하므로, 타겟과 완벽하게 동일한 인터페이스를 구현한 이 프록시 객체가 타겟을 완벽히 대체하여 클라이언트의 요청을 무리 없이 가로챌 수 있게 됩니다.

### 3. 프록시 기반 메커니즘 내에서의 공통된 역할 프록시를 생성하는 뼈대가 클래스 상속(CGLib)이든 인터페이스 구현(JDK Dynamic Proxy)이든, AOP 아키텍처 내부에서 수행하는 프록시의 역할과 통제 흐름은 동일합니다. 생성된 프록시 객체는 클라이언트의 메서드 호출을 먼저 가로채어 Pointcut 조건을 검토하고, 적용해야 할 부가 기능 로직(Advice)을 적절한 시점에 실행한 뒤, 내부적으로 진짜 타겟 객체의 비즈니스 메서드를 위임 호출하여 전체 로직을 완성합니다.

정리하면 JDK Dynamic Proxy는 프록시 기반 AOP 아키텍처에서 **"대상 객체(Target)가 인터페이스 규격을 준수할 때, 클래스 상속이 아닌 인터페이스 구현체를 동적으로 생성하여 부가 기능과 핵심 로직을 엮어내는(Weaving) 구체적인 기술적 수단"**입니다.
스프링 AOP는 타겟의 설계 구조(인터페이스 유무)에 따라 CGLib와 JDK Dynamic Proxy라는 두 가지 프록시 생성 방식을 유연하게 선택함으로써, 개발자가 작성한 원본 비즈니스 로직 코드를 전혀 훼손하지 않고도 강력하고 완벽한 관심사의 분리(Separation of Concerns)를 달성하게 해 줍니다.
