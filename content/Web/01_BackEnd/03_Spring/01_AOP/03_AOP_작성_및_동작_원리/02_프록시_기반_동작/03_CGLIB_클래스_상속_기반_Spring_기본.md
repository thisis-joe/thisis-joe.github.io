# 03_CGLIB_클래스_상속_기반_Spring_기본

## 상위 맥락
02_프록시_기반_동작 < 03_AOP_작성_및_동작_원리 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍)에서 스프링(Spring)은 타겟(Target) 객체의 원본 코드를 수정하지 않고 부가 기능을 적용(Weaving)하기 위해 프록시(Proxy) 기반의 메커니즘을 사용합니다.
이 프록시를 생성하는 방식 중 스프링의 기본 동작으로 채택된 핵심 기술이 바로 CGLIB(Code Generator Library)를 활용한 클래스 상속 기반의 프록시 생성입니다.

프록시 기반 동작이라는 더 큰 맥락 속에서 CGLIB가 갖는 구조적 특징과 역할은 다음과 같습니다.
### 1. 클래스 상속(Inheritance)을 통한 동적 프록시 생성 CGLIB는 핵심 비즈니스 로직을 보유한 타겟 클래스(예: BasicMemberDao)를 런타임에 직접 상속(extends)하여 가짜 대리 객체인 프록시를 생성합니다.

스프링 컨테이너가 AOP 적용 대상 빈을 생성할 때, 원본 타겟 클래스의 바이트코드를 조작하여 하위 클래스(Subclass) 형태의 프록시 빈을 만듭니다.

실제 테스트 코드를 확인해 보면, 이 CGLIB 프록시 객체는 BasicMemberDao$$SpringCGLIB$$0과 같이 타겟 클래스명 뒤에 $$SpringCGLIB$$가 붙는 고유한 네이밍 규칙을 갖는 것을 볼 수 있습니다.
이는 해당 객체가 단순한 원본 객체가 아니라, 스프링 CGLIB에 의해 동적으로 상속 및 확장된 대리인임을 증명합니다.

### 2. 프록시 동작 흐름 속에서의 CGLIB의 역할 클라이언트가 스프링 컨테이너에 빈을 요청(ctx.getBean())하면, 컨테이너는 원본 타겟 객체 대신 CGLIB로 생성된 하위 클래스 타입의 프록시 객체를 반환합니다.

클라이언트는 다형성에 의해 원본 타겟과 동일하게 취급되는 이 프록시 객체의 메서드를 호출하게 됩니다.

호출을 가로챈 프록시는 먼저 적용해야 할 어드바이스(Advice, 예: LoggingAspect)를 실행하여 횡단 관심사를 처리합니다.

이후 프록시는 상속 구조를 활용하여 부모 클래스인 실제 타겟의 비즈니스 메서드(예: super.method())를 내부적으로 위임 호출함으로써, 핵심 로직과 부가 기능의 완벽한 결합(Weaving)을 완성합니다.

### 3. 인터페이스 의존성 탈피 (스프링 기본 전략이 된 이유) 스프링은 자바의 기본 기능인 인터페이스 기반의 JdkDynamicProxy를 생성하는 방식도 지원합니다.
하지만 이 방식은 타겟 객체가 반드시 하나 이상의 인터페이스를 구현(implements)해야만 동작한다는 한계가 있습니다.
반면 CGLIB는 인터페이스 없이 타겟의 '클래스 자체'를 상속받아 프록시를 구성할 수 있기 때문에, 타겟 클래스 구조에 대한 어떠한 제약이나 강제성 없이 범용적으로 AOP를 적용할 수 있습니다.
이러한 강력한 유연성 덕분에 CGLIB가 스프링 부트(Spring Boot) 환경에서 AOP 프록시 생성의 기본(Default) 동작으로 자리 잡게 되었습니다.

정리하면 프록시 아키텍처 내에서 CGLIB는 "원본 코드를 손대지 않는다"는 AOP의 철학을 '객체지향의 상속(Inheritance)' 기법을 통해 가장 유연하게 풀어낸 구체적인 구현 기술입니다.
스프링은 개발자 몰래 배후에서 CGLIB를 통해 타겟을 상속받는 똑똑한 대리인(Proxy)을 생성해 줌으로써, 개발자가 인터페이스 설계에 얽매이거나 복잡한 프록시 패턴을 직접 구현할 필요 없이 오직 비즈니스 로직 작성에만 집중할 수 있는 완벽한 추상화를 제공합니다.
