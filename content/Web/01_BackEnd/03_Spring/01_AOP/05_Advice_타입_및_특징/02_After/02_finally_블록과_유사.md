# 02_finally_블록과_유사

## 상위 맥락
02_After < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(관점 지향 프로그래밍) 아키텍처에서 @After 어드바이스(Advice)는 자바 예외 처리 구문의 'finally 블록과 유사'한 구조적 역할을 수행하며, 타겟(Target) 메서드의 실행 결과와 무관하게 부가 기능을 반드시 보장해야 할 때 사용되는 핵심적인 제어 메커니즘입니다.
다양한 어드바이스 타입이 존재하는 AOP의 더 큰 맥락 속에서 @After가 갖는 'finally 블록과의 유사성'과 그 원리를 단계별로 분석하면 다음과 같습니다.
### 1. @After 어드바이스의 실행 시점과 무조건적 보장 AOP에서 어드바이스는 조인 포인트(메서드 실행)를 기준으로 부가 기능이 개입할 타이밍을 결정합니다. 이때 @After는 타겟 메서드의 실행이 완전히 종료된 직후에 개입합니다. 가장 중요한 특징은 타겟 메서드가 정상적으로 로직을 완수하여 값을 반환(return)하든, 혹은 내부적인 문제로 인해 예외를 던지며(throws Exception) 비정상 종료되든 그 성공 여부와 철저하게 무관하게 언제나 부가 기능의 실행을 보장한다는 점입니다.

### 2. try ~ catch ~ finally 구조와의 논리적 매핑 이러한 무조건적인 실행 보장 속성 때문에 @After 어드바이스의 동작 원리는 자바의 전통적인 예외 처리 문법인 try ~ catch ~ finally 블록 중 finally 블록의 역할에 정확히 해당합니다.
기존의 객체지향(OOP) 방식에서는 시스템 리소스(예: DB 접속 관리)를 반환하거나 필수적인 종료 로그를 남기기 위해 핵심 비즈니스 로직을 try로 감싸고, 실패 시 catch를 거친 후, 무슨 일이 있어도 실행되어야 하는 정리 코드를 finally 내부에 직접 하드코딩해야만 했습니다.

### 3. 제어 흐름의 추상화 및 아키텍처적 가치 AOP는 이 finally 블록이 담당하던 '의무적인 사후 처리 로직'의 제어 흐름을 @After라는 선언적인 애너테이션으로 완벽하게 추상화해 냅니다.
만약 정상 종료 시에만 필요한 로직이라면 @AfterReturning을, 예외 발생 시에만 필요한 대책 수립이라면 @AfterThrowing을 분리해서 사용하게 됩니다.

하지만 @After는 이 둘을 모두 포괄하는 finally의 속성을 지니므로, 프록시(Proxy)는 타겟의 실행 결과를 가로챈 뒤 그 결과의 상태를 따지지 않고 내부적으로 지정된 공통 사후 처리(예: 리소스 해제, 세션 종료 등)를 즉각 수행합니다.

정리하면 @After 어드바이스 내의 'finally 블록과 유사'하다는 원리는 핵심 비즈니스 로직의 코드를 에러 핸들링을 위한 보일러플레이트(Boilerplate) 코드로부터 완벽하게 해방시키는 AOP의 주요 성과입니다.
개발자는 비즈니스 로직 내부에 복잡한 try ~ finally 구조를 명시할 필요 없이 순수한 도메인 로직 작성에만 집중할 수 있으며, 시스템의 안정성을 위해 무조건적으로 실행되어야 하는 횡단 관심사(리소스 정리 등)는 @After를 통해 프록시 계층에서 일관되고 안전하게 직조(Weaving)되는 고도화된 모듈화 아키텍처가 완성됩니다.
