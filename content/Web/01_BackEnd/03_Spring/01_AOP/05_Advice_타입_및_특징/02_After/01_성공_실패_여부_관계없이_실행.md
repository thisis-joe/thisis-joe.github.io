# 01_성공_실패_여부_관계없이_실행

## 상위 맥락
02_After < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(Spring AOP) 아키텍처에서 조인 포인트(Join point)에 부가 기능이 개입하는 시점을 정의하는 어드바이스(Advice) 타입 중 하나인 @After는, 핵심 비즈니스 로직(Target)의 실행 종료 시점에 작동하는 횡단 관심사입니다.
이러한 @After의 맥락 속에서 '성공/실패 여부 관계없이 실행'된다는 것은 시스템의 제어 흐름과 안정성을 보장하는 매우 중요한 구조적 특징을 의미하며, 구체적인 내용은 다음과 같습니다.
### 1. 정상 종료와 예외 발생 상황 모두를 포괄하는 실행 보장 @After 어드바이스의 가장 핵심적인 동작 원리는 타겟 메서드가 결괏값을 반환하며 정상적으로 종료(return)되었든, 혹은 내부 에러로 인해 예외를 던지며 실패(throws Exception)했든 그 여부와 무관하게 언제나 실행이 보장된다는 점입니다.
즉, 비즈니스 로직의 결과 상태가 프록시(Proxy)의 부가 기능 수행을 방해할 수 없도록 설계되어 있습니다.
### 2. 자바의 finally 블록에 해당하는 역할 이러한 절대적인 실행 보장 특성을 자바(Java) 언어의 기본 문법에 빗대어 보면, try ~ catch ~ finally 예외 처리 구문에서 예외 발생 여부와 상관없이 무조건 마지막에 호출되는 finally 블록의 역할과 정확히 일치합니다.
내부 동작 흐름도를 보더라도, 프록시는 타겟의 비즈니스 로직 실행 구역에서 예외가 발생하든 정상 리턴이 발생하든, 제어권이 프록시로 돌아오면 예외를 클라이언트로 던지기 직전이나 값을 반환하기 직전에 반드시 @After 어드바이스를 수행합니다.

정리하면 @After의 '성공/실패 여부 관계없이 실행'되는 특성은 시스템의 자원 누수(Resource Leak)를 방지하고 후처리를 완결 짓기 위한 필수적인 안전장치입니다.
개발자는 이 특성을 활용하여 데이터베이스 커넥션 반환, 스레드 락(Lock) 해제, 임시 파일 삭제 등 비즈니스 로직이 중간에 실패하더라도 시스템의 안정성을 위해 무조건적으로 수행되어야 하는 '리소스 정리(Cleanup)' 및 필수 감사 로깅 등의 횡단 관심사를 타겟 객체로부터 완벽하게 분리해 낼 수 있습니다.
이를 통해 비즈니스 코드 내부에 지저분한 finally 블록을 반복적으로 작성하지 않고도, 선언적인 AOP 설정만으로 견고한 시스템 아키텍처를 유지할 수 있게 됩니다.
