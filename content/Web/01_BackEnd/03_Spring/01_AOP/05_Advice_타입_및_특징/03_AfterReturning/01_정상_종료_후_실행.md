# 01_정상_종료_후_실행

## 상위 맥락
03_AfterReturning < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 생태계에서 어드바이스(Advice)는 타겟(Target) 메서드의 실행 흐름에 개입하는 시점에 따라 세분화됩니다.
그중 @AfterReturning은 타겟 메서드의 실행 과정 중 '정상 종료(return)'라는 매우 엄격하고 구체적인 상태 변화를 트리거(Trigger)로 삼아 동작하는 후처리 개입 지점입니다.

@AfterReturning 아키텍처 내 '정상 종료 후 실행'이 갖는 구조적 의미와 내부 동작 인과관계를 분석하면 다음과 같습니다.
### 1. '정상 종료 후 실행'의 아키텍처적 의미와 제어 흐름 @AfterReturning은 타겟 메서드가 로직을 수행하는 도중 어떠한 예외(Exception)도 던지지 않고, 부여된 핵심 비즈니스 로직을 끝까지 완수하여 성공적으로 결과값을 반환(Return)했을 때에만 제한적으로 동작합니다.

실행 흐름의 인과관계: 클라이언트의 요청이 프록시(Proxy)를 거쳐 타겟으로 위임된 후, 타겟 메서드가 처리를 마치면 제어권은 다시 프록시로 돌아옵니다.
이때 프록시는 타겟의 실행 결과를 평가하며, 만약 타겟 내부에서 예외가 발생하여 비정상 종료되었다면 @AfterReturning 로직은 시퀀스 상에서 완전히 무시(Skip)되고 개입하지 않습니다.
오직 '성공적인 완료'가 보장된 상태에서만 동작하도록 설계된 조건부 어드바이스입니다.
### 2. 반환값(Return Value)의 획득 및 바인딩 메커니즘 이 어드바이스의 가장 핵심적인 기능은 타겟이 생산해 낸 결과물(반환값)을 클라이언트에게 최종 전달하기 직전에 프록시가 가로채어 읽어 들일 수 있다는 것입니다.

이를 위해 @AfterReturning 애너테이션 내부에는 returning이라는 속성을 명시합니다.

개발자는 이 returning 속성에 타겟의 반환값을 저장할 '변수 명'을 지정하며, 이 이름은 반드시 어드바이스 메서드가 전달받는 파라미터의 이름과 정확히 일치해야 프록시가 런타임에 해당 결과값을 매핑(Binding)해 줄 수 있습니다.

### 3. 획득한 결과값의 후속 조작 (Data Manipulation) 정상 종료 후 전달받은 반환값은 단순한 조회를 넘어 **프록시 계층에서의 데이터 조작(Manipulation)**이 가능합니다.

제약 사항: 타겟이 반환한 객체 자체를 아예 다른 새로운 객체 인스턴스로 '완전 대체(Replace)'하는 것은 이 어드바이스의 권한 밖이므로 불가능합니다.
(완전한 대체는 @Around 어드바이스에서만 가능합니다.)
상태 변경의 허용: 그러나 반환된 값이 레퍼런스(참조) 타입의 객체일 경우, 해당 객체 내부에 접근하여 속성(Property) 값을 변경하는 것은 얼마든지 가능합니다.

적용 예시: 비즈니스 로직(Target)이 데이터베이스에서 조회한 Member 객체를 정상적으로 반환했을 때, @AfterReturning 어드바이스가 이 객체를 낚아채어 클라이언트에게 전달되기 전 member.setPassword("*")와 같이 암호 필드를 마스킹(Masking) 처리하는 보안 로직을 수행할 수 있습니다.

정리하면 @AfterReturning에서의 '정상 종료 후 실행' 메커니즘은, 핵심 비즈니스 로직의 "성공적인 처리"를 전제로 한 데이터의 사후 가공(Formatting, Masking)이나 성공에 대한 후속 감사 로그(Audit Logging) 등의 횡단 관심사를 순수 비즈니스 코드로부터 완벽하게 분리해 내는 강력한 수단입니다.
개발자는 이 구조를 통해 핵심 코드는 본연의 역할(데이터의 생성 및 조회)에만 집중하게 놔두고, 성공 시의 부가적인 데이터 조작은 프록시 영역에서 투명하게 엮어내는(Weaving) 고도의 결합도 감소를 이루어낼 수 있습니다.
