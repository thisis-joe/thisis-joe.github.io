# 02_리턴_값_조작_및_접근_가능

## 상위 맥락
03_AfterReturning < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 아키텍처에서 @AfterReturning 어드바이스(Advice)는 핵심 비즈니스 로직(Target)이 예외 없이 성공적으로 실행을 마치고 결과값을 반환(Return)하는 시점에 개입하는 사후 처리 모듈입니다.
이러한 @AfterReturning의 가장 강력하고 특징적인 메커니즘이 바로 **'리턴 값 조작 및 접근 기능'**이며, 프록시(Proxy) 기반의 제어 흐름 속에서 이 기능이 어떻게 동작하고 어떤 한계를 지니는지 구체적인 인과관계를 분석하면 다음과 같습니다.
### 1. 리턴 값 접근(Accessibility)의 바인딩 원리 프록시는 타겟 메서드의 실행이 정상적으로 완료되면, 타겟이 클라이언트에게 돌려주려던 '반환값(Return value)'을 중간에서 먼저 획득합니다. 이 값을 어드바이스 로직 내부로 끌고 들어오기 위해 스프링 AOP는 다음과 같은 엄격한 바인딩 규칙을 요구합니다.
@AfterReturning 애너테이션 내부에는 returning이라는 속성을 명시해야 하며, 여기에 타겟의 반환값을 저장할 '변수명'을 지정합니다.

이때 선언된 변수명은 반드시 실제 어드바이스 메서드가 전달받는 파라미터의 이름과 완벽하게 일치해야 합니다.
(예: returning = "member"로 지정했다면, 메서드 파라미터 역시 Member member 형태로 선언되어야 함
)
이러한 인과관계를 통해 프록시는 런타임에 타겟의 반환값을 어드바이스 메서드의 인자로 정확하게 주입(Injection)하여 접근을 허용합니다.

### 2. 리턴 값 조작(Manipulation)의 허용 범위와 엄격한 한계 어드바이스 내부로 전달받은 반환값은 비즈니스 요구사항에 맞게 사후 조작이 가능하지만, 객체 지향의 참조(Reference) 원리에 따라 명확한 한계가 존재합니다.
완전 대체 불가: 프록시가 이미 획득한 원본 객체 자체를 아예 다른 새로운 객체 인스턴스로 통째로 교체(대체)하여 클라이언트에게 반환하는 것은 아키텍처 상 불가능합니다.

객체 내부 속성(Property) 변경 가능: 반면, 반환된 값이 레퍼런스(참조) 타입의 객체일 경우, 해당 객체의 메모리 주소를 참조하여 객체 내부의 속성(상태 값)을 변경하는 조작은 완벽하게 허용됩니다.

### 3. 적용 사례를 통한 아키텍처적 의의 (데이터 마스킹) '비밀번호 마스킹' 사례는 이 원리를 가장 잘 보여주는 실무적 패턴입니다.
데이터베이스에서 조회된 Member 객체(Target의 반환값)가 클라이언트에게 그대로 노출되기 직전에, @AfterReturning 어드바이스가 이를 가로채어 파라미터로 전달받습니다.
이후 member.setPassword("*")와 같이 객체 내부의 민감한 속성값만을 안전하게 조작(마스킹)한 뒤 제어권을 넘김으로써, 원본 비즈니스 코드를 전혀 훼손하지 않고도 시스템의 보안 요건을 충족시킬 수 있습니다.

정리하면 @AfterReturning 컨텍스트 내에서 '리턴 값 조작 및 접근' 기능은 핵심 비즈니스 로직이 생산해 낸 결과물(Data)이 시스템 외부(클라이언트)로 응답되기 직전에 거쳐야 하는 '출구 필터(Exit Filter)' 역할을 수행합니다.
개발자는 이 메커니즘을 통해 객체의 완전 대체라는 파괴적인 조작은 방지하면서도(안전성 보장), 반환되는 데이터의 포맷팅, 민감 정보의 마스킹, 사후 감사 로그 작성 등 데이터의 내부 속성을 가공하는 고도화된 사후 처리(Post-processing)를 관점 지향적으로 완벽하게 모듈화할 수 있습니다.
