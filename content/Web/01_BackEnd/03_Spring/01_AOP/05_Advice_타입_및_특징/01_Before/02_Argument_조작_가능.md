# 02_Argument_조작_가능

## 상위 맥락
01_Before < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 아키텍처에서 @Before 어드바이스(Advice)는 클라이언트의 호출이 실제 타겟(Target) 메서드로 진입하기 직전에 제어권을 획득하여 선행 작업을 수행하는 부가 기능 모듈입니다.
이 단계에서 제공되는 '전달받은 Argument 조작 가능' 기능은 데이터 전처리나 암호화 같은 횡단 관심사를 핵심 비즈니스 로직과 분리해 내는 데 중추적인 역할을 수행합니다.
@Before 어드바이스의 전체적인 동작 흐름과 인과관계 속에서 파라미터(Argument) 조작의 원리를 심도 있게 분석하면 다음과 같습니다.
### 1. @Before의 동작 시점과 제어 흐름 클라이언트가 프록시(Proxy)에 요청을 보내면, 프록시는 내부의 실제 타겟 로직을 호출하기 전에 @Before 어드바이스에 정의된 부가 기능을 먼저 실행시킵니다.
이 선행 작업이 정상적으로 종료되어야만 비로소 타겟의 비즈니스 메서드로 제어권이 넘어가며, 만약 @Before 로직 수행 중 예외(Exception)가 발생한다면 타겟 메서드는 아예 실행되지 못하고 전체 호출 흐름이 중단됩니다.

### 2. Argument(파라미터) 조작의 범위와 한계 (원리) @Before는 클라이언트가 타겟 메서드를 실행하기 위해 전달한 파라미터(Argument)를 중간에서 확보하여 검사하거나 변형할 수 있는 권한을 가집니다.

객체 속성(Property) 변경 가능: 파라미터로 전달된 값이 레퍼런스(참조) 타입의 객체일 경우, 어드바이스 내부에서 해당 객체의 상태나 속성 값을 직접 변경(Setter 호출 등)할 수 있습니다.

완전 대체 불가: 하지만 @Before의 구조적 한계상, 전달받은 파라미터 인스턴스 자체를 메모리상의 완전히 다른 새로운 객체로 대체해 버리는 것은 불가능합니다.
(파라미터의 완전 대체나 반환값 조작 등 메서드 실행의 모든 것을 제어하려면, 타겟 호출을 직접 통제하는 @Around 어드바이스를 사용해야 합니다
).
### 3. args 지정자를 통한 파라미터 바인딩과 조작 실무 (인과관계) 어드바이스가 파라미터를 조작하기 위해서는 먼저 그 데이터에 접근해야 합니다. 이때 JoinPoint.getArgs()를 호출하여 Object 배열에서 값을 추출하는 원시적인 방법 대신, 포인트컷(Pointcut) 표현식의 args(변수명)을 사용하여 타겟으로 향하던 파라미터를 어드바이스 메서드의 인자로 편리하게 직접 바인딩(매핑)받을 수 있습니다.

적용 메커니즘 (비밀번호 암호화 예시): 구현 사례에서는 회원 가입 로직을 수행하는 save 메서드 호출 전 @Before("execution(...) && args(member)")를 선언하여 클라이언트가 넘긴 Member 객체를 낚아챕니다.
이후 어드바이스 로직 내에서 member.setPassword(...)를 실행하여 평문 비밀번호를 암호화된 상태로 변환(조작)합니다.

흐름의 완성: 이렇게 @Before 단계에서 파라미터(객체 속성) 조작이 성공적으로 완료되면, 프록시는 이 변형된 Member 객체를 그대로 들고 실제 타겟의 save 비즈니스 로직으로 위임(전달)합니다.
결과적으로 타겟 객체는 자신이 암호화를 수행하지 않았음에도 불구하고 안전하게 가공된 데이터를 받아 DB에 저장하게 됩니다.

정리하면 @Before 어드바이스 생태계 내에서 'Argument 조작 가능' 특성은 핵심 비즈니스 로직의 코드를 훼손하지 않은 채, 메서드 실행 직전에 입력 데이터를 검증하거나 규격화(가공, 포맷 변환, 암호화)하는 전처리(Pre-processing) 작업을 수행하기 위한 필수적인 메커니즘입니다.
객체의 레퍼런스를 통해 내부 상태만을 정교하게 변경하는 이 방식 덕분에, 대상 비즈니스 로직(Target)은 외부의 개입을 전혀 의식하지 않고도 순수성을 완벽히 유지할 수 있습니다.
