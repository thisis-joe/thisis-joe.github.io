# 01_Target_메서드_호출_전_실행

## 상위 맥락
01_Before < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 생태계에서 @Before 어드바이스(Advice)는 클라이언트의 요청이 실제 비즈니스 로직(Target)에 도달하기 직전에 횡단 관심사를 선제적으로 처리하는 가장 대표적인 개입 지점입니다.

'Target 메서드 호출 전 실행'이라는 동작 특성이 프록시(Proxy) 아키텍처라는 더 큰 맥락 속에서 어떻게 구현되고 어떤 강력한 제어권을 행사하는지 관점에서 구체적인 인과관계를 살펴보면 다음과 같습니다.
### 1. 프록시를 통한 선행 제어 및 타겟 호출 차단(Block) 권한 클라이언트가 타겟의 메서드를 호출하면, 중간에 개입한 프록시(Proxy)는 내부적으로 진짜 타겟 객체에게 로직을 위임하기 전에 @Before 로직을 최우선으로 실행시킵니다.
이 흐름에서 가장 중요한 아키텍처적 특징은, 만약 @Before 어드바이스 실행 도중 예외(Exception)가 발생하게 되면 그 즉시 시스템의 제어 흐름이 중단되어 실제 Target 메서드는 아예 동작조차 할 수 없게 된다는 점입니다.
이는 핵심 비즈니스 로직이 실행되기 전에 반드시 보장되어야 하는 조건(예: 보안 인증 및 권한 검사)을 완벽하게 통제하고 타겟을 보호할 수 있음을 의미합니다.
### 2. 전달 파라미터(Argument)의 사전 획득 및 데이터 조작 @Before는 타겟이 업무를 시작하기 '전'에 동작하므로, 클라이언트가 타겟 메서드로 전달하려 했던 파라미터(인자값)를 프록시 단계에서 미리 가로채어 열어보고 조작할 수 있습니다.

이때 파라미터 객체 자체를 다른 새로운 객체로 완전히 대체하는 것은 불가능합니다.

하지만, 전달받은 인자가 레퍼런스(참조) 객체일 경우, 해당 객체 내부의 속성 값을 변경하는 것은 얼마든지 가능합니다.

회원 저장(save) 메서드 실행 전에 @Before가 개입하여 Pointcut의 args(member)를 통해 파라미터로 넘어온 Member 객체를 직접 참조하고, 객체 내부의 비밀번호 값을 추출해 암호화(조작)한 뒤 다시 세팅(member.setPassword(...))하는 사전 데이터 가공 프로세스를 거칩니다.

정리 결과적으로 @Before 컨텍스트에서의 'Target 메서드 호출 전 실행'은 단순한 시점의 앞섬을 넘어, 핵심 비즈니스 로직으로 진입하기 위한 '사전 검문소' 역할을 완벽히 수행합니다.
개발자는 이 메커니즘을 통해 타겟 메서드가 실행되기 전에 파라미터의 유효성을 검증하거나 데이터를 사전 가공(암호화 등)할 수 있으며, 조건에 맞지 않을 경우 예외를 발생시켜 원본 타겟의 실행을 원천 차단함으로써, 핵심 비즈니스 로직(Target)이 언제나 가장 안전하고 검증된 상태에서만 수동적으로 실행되도록 강력한 의존성 제어 체계를 구축할 수 있습니다.
