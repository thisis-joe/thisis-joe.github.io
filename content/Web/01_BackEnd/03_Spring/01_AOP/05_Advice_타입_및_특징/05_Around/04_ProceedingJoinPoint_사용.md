# 04_ProceedingJoinPoint_사용

## 상위 맥락
05_Around < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 아키텍처에서 @Around 어드바이스는 타겟(Target) 메서드 실행의 이전과 이후 흐름 전체에 대한 제어권을 완벽하게 장악하는 가장 강력한 횡단 관심사 모듈입니다.
이러한 전면적인 제어력을 시스템적으로 구현하기 위해 어드바이스 메서드가 반드시 파라미터로 전달받아야 하는 핵심 매개체가 바로 **ProceedingJoinPoint**입니다.

@Around의 거시적 맥락 속에서 ProceedingJoinPoint 객체의 구체적인 사용 방식과 그로 인해 발생하는 아키텍처적 인과관계는 다음과 같습니다.
### 1. proceed() 메서드를 통한 명시적 타겟 호출 제어 다른 어드바이스(예: @Before, @After)들이 정해진 시점에 프록시(Proxy)에 의해 수동적으로 실행되는 것과 달리, @Around 내에서는 ProceedingJoinPoint가 제공하는 proceed() 메서드를 개발자가 명시적으로 호출해야만 실제 타겟의 핵심 비즈니스 로직이 동작합니다.
이러한 구조적 특징은 개발자에게 실행 흐름에 대한 절대적인 통제권을 부여합니다.
즉, 타겟 로직의 실행 시점을 임의로 조절할 수 있음은 물론, 상황(예: 이미 캐싱된 데이터가 존재하는 경우)에 따라 아예 proceed()를 호출하지 않음으로써 타겟 메서드의 실행 자체를 원천적으로 차단(Bypass)하는 극단적인 제어까지 가능해집니다.

### 2. 파라미터(Argument)의 완전한 대체(Replacement) ProceedingJoinPoint는 클라이언트가 타겟으로 전달하려 했던 원본 파라미터를 조작하는 데 있어 가장 강력한 수준의 권한을 제공합니다. @Before 어드바이스가 파라미터 레퍼런스의 내부 속성만을 변경할 수 있었던 반면, @Around에서는 ProceedingJoinPoint.getArgs()를 통해 파라미터 배열을 추출한 뒤, 그 값 자체를 완전히 새로운 값이나 인스턴스로 덮어씌워 완전 대체(Replacement)하는 것이 가능합니다.
이렇게 재구성된 파라미터 배열은 오버로딩된 proceed(args) 메서드의 인자로 전달되며, 결국 타겟 객체는 원본이 아닌 프록시가 새롭게 규정한 입력값을 바탕으로 로직을 수행하게 됩니다.

### 3. 실행 결과(Return Value)의 가로채기 및 조작 타겟 메서드의 실행이 성공적으로 완료되면, proceed() 메서드는 타겟이 생성한 최종 결괏값을 반환합니다.
이때 @Around 어드바이스는 이 반환값을 클라이언트에게 즉시 넘기지 않고 변수에 담아둔 뒤, 추가적인 연산을 수행하거나 완전히 다른 새로운 결과값으로 조작하여 최종 반환할 수 있습니다.
실제 적용 예시를 보면, proceed()를 통해 얻은 정수 결과값에 대해 조건부 연산(result % 2 == 0 ? result : result / 2)을 수행한 뒤 전혀 다른 값으로 탈바꿈시켜 반환하는 데이터 후처리 과정을 확인할 수 있습니다.

정리하면 @Around 생태계 내에서 ProceedingJoinPoint의 사용은 핵심 비즈니스 로직(Target)의 3대 생명주기인 '입력(파라미터 대체)', '실행(호출 여부 및 타이밍 결정)', '출력(반환값 조작)'에 대한 모든 제어권을 프록시 계층으로 이관받기 위한 핵심적인 객체 지향 계약입니다.

개발자는 이 객체를 활용함으로써 비즈니스 코드를 단 한 줄도 훼손하지 않은 채, 파라미터의 완전한 규격 변환, 고도화된 캐싱(Caching) 전략, 예외 발생 시의 재시도(Retry) 로직 구현 등 AOP가 제공할 수 있는 가장 복잡하고 정밀한 형태의 관심사 분리(Separation of Concerns) 아키텍처를 완성할 수 있습니다.
