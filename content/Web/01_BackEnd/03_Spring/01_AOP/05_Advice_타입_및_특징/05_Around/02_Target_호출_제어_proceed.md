# 02_Target_호출_제어_proceed

## 상위 맥락
05_Around < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 어드바이스(Advice) 생태계에서 @Around는 타겟(Target) 메서드 실행의 이전과 이후, 그리고 예외 발생 시점까지의 모든 흐름을 단일 모듈 내에서 완벽하게 장악하는 가장 강력한 어드바이스 타입입니다.

이러한 @Around의 거시적 맥락 속에서 'Target 호출 제어 (proceed)'는 프록시(Proxy)가 핵심 비즈니스 로직의 실행 권한을 능동적으로 쥐고 통제하는 핵심 메커니즘을 의미합니다.
이 제어 메커니즘의 원리와 아키텍처적 파급력을 단계별로 분석하면 다음과 같습니다.
### 1. ProceedingJoinPoint와 실행 흐름의 수동적 위임 @Before나 @After와 같은 다른 어드바이스들은 프록시가 내부적으로 타겟 메서드를 자동 호출하는 흐름 속에서 특정 시점에 '기생'하여 동작합니다. 그러나 @Around 어드바이스는 파라미터로 반드시 ProceedingJoinPoint 타입의 객체를 전달받으며, 개발자가 어드바이스 내부에서 proceed() 메서드를 명시적으로 호출해야만 비로소 실제 Target의 메서드가 실행됩니다.
즉, 핵심 비즈니스 로직을 언제 실행할지, 혹은 아예 실행하지 않을지(예: 캐시에서 데이터를 반환하여 타겟 호출 생략)를 프록시 단계에서 완벽하게 통제할 수 있는 절대적인 제어권을 확보하게 됩니다.
### 2. 파라미터(Argument)의 완전한 대체 (Pre-processing) Target 메서드를 호출(proceed())하기 전, @Around는 클라이언트가 전달한 원본 파라미터를 획득하고 이를 완전히 다른 값으로 대체할 수 있습니다.
원리: ProceedingJoinPoint.getArgs()를 통해 원본 인자 배열을 가져온 뒤, 특정 인덱스의 값을 새롭게 연산(예: args * 10)하여 조작합니다.

적용: 이후 오버로딩된 proceed(args) 메서드에 이 조작된 배열을 파라미터로 넘겨주면, Target 객체는 클라이언트가 보낸 원본이 아닌 프록시가 가공한 새로운 데이터를 기반으로 비즈니스 로직을 수행하게 됩니다.
이는 속성 값만 변경 가능했던 @Before의 한계를 뛰어넘는 '파라미터 완전 대체' 기능입니다.

### 3. 리턴 값(Return Value)의 조작 및 완전 대체 (Post-processing) Target 메서드의 호출 결과 역시 @Around의 통제하에 놓입니다.
원리: proceed()가 실행된 후 반환하는 결과값을 변수(예: Integer result)로 할당받습니다.

적용: 어드바이스는 이 결과값을 그대로 클라이언트에게 돌려줄 수도 있지만, 비즈니스 요구에 따라 값을 재연산(예: result / 2)하거나 완전히 다른 타입의 객체로 포장하여 반환할 수도 있습니다.
이 역시 반환된 객체의 참조를 통한 속성 변경만 가능했던 @AfterReturning과 비교해 훨씬 강력한 수준의 데이터 후처리 능력을 제공합니다.

정리하면 @Around 환경에서의 **proceed() 호출 제어는 타겟(Target) 객체를 프록시라는 거대한 컨테이너 내부에 완벽히 캡슐화(Encapsulation)하는 아키텍처적 경계(Boundary)**입니다.
개발자는 proceed()를 중심으로 상단에는 보안 검증이나 파라미터 암호화(조작)를, 하단에는 결과값 마스킹이나 트랜잭션 커밋 처리를 구현할 수 있으며, 호출 자체를 try-catch로 감싸 예외를 복구하는 등 자바의 모든 프로그래밍적 제어를 적용할 수 있습니다.
이를 통해 단일 어드바이스 내에서 캐싱(Caching), 성능 측정(Profiling), 트랜잭션(Transaction) 관리와 같은 고도화된 횡단 관심사를 매우 직관적이고 강력하게 구현할 수 있게 됩니다.
