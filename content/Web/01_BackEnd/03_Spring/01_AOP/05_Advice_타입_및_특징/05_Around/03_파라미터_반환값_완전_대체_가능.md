# 03_파라미터_반환값_완전_대체_가능

## 상위 맥락
05_Around < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP의 5가지 어드바이스(Advice) 타입 중 @Around는 타겟(Target) 메서드의 실행 이전과 이후 모든 시점을 단일 메서드 내에서 완벽하게 장악하는 가장 강력한 제어 모듈입니다.

이전 단계의 어드바이스들(@Before, @AfterReturning)이 타겟으로 향하는 객체나 반환되는 객체의 내부 속성(Property)만을 제한적으로 변경할 수 있었던 반면, @Around는 파라미터(Argument)와 리턴(반환) 값 모두를 메모리상의 새로운 값으로 '완전하게 대체(Replace)'할 수 있는 독보적인 제어권을 갖습니다.

이 완벽한 대체의 원리와 인과관계를 단계별로 분석하면 다음과 같습니다.
### 1. 제어권의 장악 원리: ProceedingJoinPoint와 proceed() 이러한 완전 대체가 가능한 시스템적 이유는 @Around가 타겟 호출의 주도권을 직접 행사하기 때문입니다. 다른 어드바이스들과 달리 @Around는 파라미터로 반드시 ProceedingJoinPoint를 전달받으며, 내부에서 개발자가 명시적으로 proceed() 메서드를 호출해야만 실제 타겟 비즈니스 로직이 실행됩니다.
즉, proceed() 호출 전은 사전 제어 구역이 되고, 호출 후는 사후 제어 구역이 되는 인과관계를 가집니다.
### 2. 파라미터(Argument) 완전 대체 메커니즘
클라이언트가 요청 시 넘긴 파라미터들은 타겟에 도달하기 전 pjp.getArgs()를 통해 프록시 배열 형태로 획득됩니다.

개발자는 이 배열 안의 값을 객체의 속성 변경 수준이 아니라, 완전히 새로운 값으로 연산하여 교체할 수 있습니다.
예를 들어, 인자로 들어온 정수 값에 10을 곱하여 배열을 업데이트(args = (Integer) args * 10;)하는 식의 완전한 조작이 성립합니다.

이후 변경된 배열을 인자로 담아 pjp.proceed(args)를 호출하게 되면, 타겟 객체는 클라이언트가 전달한 원본 데이터가 아닌 프록시가 완전히 대체해버린 새로운 파라미터를 기반으로 핵심 로직을 수행하게 됩니다.

### 3. 반환값(Return Value) 완전 대체 및 결과 조작 메커니즘
pjp.proceed()가 실행을 마치면, 타겟 객체가 생성한 결과값이 클라이언트로 곧바로 응답되지 않고 어드바이스 내의 변수(예: result)로 수신됩니다.

@Around는 이 원본 반환값을 바탕으로 새로운 연산을 수행하거나 캐싱된 다른 데이터를 가져와, 타겟이 만들어낸 결과와 완전히 다른 형태나 값으로 대체하여 최종 return 할 수 있습니다.
사례처럼 결과값이 짝수가 아닐 경우 2로 나누어 전혀 다른 정숫값을 반환하는 등 결과 데이터 흐름을 프록시 계층에서 완벽히 재가공합니다.

정리하면 @Around 생태계에서 '파라미터/반환값 완전 대체 가능' 속성은 타겟 객체의 코드를 한 줄도 훼손하지 않으면서 입력 데이터의 규격화와 출력 데이터의 전면적인 가공(캐싱, 암호화, 포맷 변환 등)을 프록시가 독점적으로 수행할 수 있게 해주는 핵심 메커니즘입니다.
개발자는 이 강력한 제어력을 통해 타겟 메서드의 실행 환경 자체를 완벽하게 통제하고 캡슐화하는 고도의 관심사 분리를 달성할 수 있습니다.
