# 01_가장_강력한_Advice

## 상위 맥락
05_Around < 05_Advice_타입_및_특징 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(관점 지향 프로그래밍) 생태계에서 @Around는 타겟(Target) 메서드의 실행 이전과 이후, 그리고 예외 발생 시점까지의 모든 생명주기를 단일 어드바이스 내에서 완벽하게 캡슐화하는 가장 강력한(Most Powerful) 어드바이스 타입입니다.

이러한 @Around의 맥락 속에서 '가장 강력한 Advice'라는 수식어는 프록시(Proxy)가 타겟의 단순한 보조자를 넘어, 타겟의 실행 흐름과 입출력 데이터 전체를 '완전하게 통제하고 대체(Replace)할 수 있는 독점적 제어권'을 갖는다는 아키텍처적 사실을 의미합니다.
이 강력함의 원리와 인과관계를 3가지 핵심 메커니즘으로 분석하면 다음과 같습니다.
### 1. proceed()를 통한 실행 흐름의 능동적 지배권 다른 어드바이스들(@Before, @After 등)은 타겟 메서드가 실행되는 과정 중 특정 시점에 수동적으로 호출되어 부가 기능만을 수행합니다.
그러나 @Around는 동작 원리가 근본적으로 다릅니다.
@Around는 파라미터로 반드시 ProceedingJoinPoint 객체를 전달받으며, 어드바이스 내부에서 개발자가 직접 proceed() 메서드를 호출해야만 비로소 실제 타겟(Target)의 핵심 비즈니스 로직이 실행됩니다.
이는 역으로 말해, @Around 내부의 조건 로직에 따라 proceed()를 아예 호출하지 않기로 결정한다면 타겟 메서드의 실행 자체를 원천적으로 차단(Short-circuit)해버릴 수 있는 능동적인 지배권을 의미합니다.
### 2. 입력 파라미터(Argument)의 완전한 대체(Replacement) @Before 어드바이스 역시 타겟 메서드 호출 전에 파라미터를 가로채어 객체 내부의 속성을 변경할 수는 있었지만, 파라미터 객체 자체를 다른 인스턴스로 완전히 교체하는 것은 불가능했습니다.
하지만 @Around는 proceed(Object[] args)를 호출할 때 조작된 파라미터 배열을 직접 넘겨주는 방식을 취하므로, 클라이언트가 전달한 원본 파라미터를 완전히 다른 값이나 새로운 객체로 100% 대체하여 타겟에게 주입할 수 있습니다.
예시 코드에서는 원본 파라미터 args에 10을 곱한 완전히 새로운 값을 proceed()의 인자로 전달하여 타겟의 동작 환경을 근본적으로 재구성하는 것을 확인할 수 있습니다.

### 3. 결과 반환값(Return Value)의 완전한 대체 @AfterReturning 어드바이스는 타겟이 반환한 객체를 참조하여 속성을 조작할 수는 있어도, 클라이언트에게 돌아가는 반환값 자체를 다른 타입이나 아예 다른 객체로 바꿔치기할 수는 없습니다.
반면 @Around는 proceed() 호출의 결과로 반환된 타겟의 원본 결과값을 어드바이스 내부 지역 변수(result)로 먼저 할당받습니다.
이후 이 원본 반환값을 기반으로 특정 연산을 수행하거나 캐시(Cache)된 완전히 다른 객체를 생성하여 클라이언트에게 최종 return 함으로써 결과값을 완벽하게 조작 및 대체할 수 있습니다.

정리하면 @Around가 가장 강력한 Advice로 평가받는 이유는, 단일 메서드 컨텍스트 내에서 '사전 처리(파라미터 조작) -> 타겟 실행 제어 -> 사후 처리(결과값 조작) 및 예외 제어'라는 횡단 관심사의 모든 시나리오를 유기적으로 연결하여 상태를 공유할 수 있기 때문입니다.
이러한 구조적 특성 덕분에 @Around는 단순한 로깅을 넘어, 실행 전후의 시간을 측정하여 성능을 프로파일링(Performance Profiling)하거나, 데이터베이스 트랜잭션의 시작과 커밋/롤백을 하나의 블록 안에서 관리하고, 기존 결과를 가로채어 캐싱(Caching) 시스템을 구현하는 등, 시스템 아키텍처의 가장 복잡하고 핵심적인 횡단 관심사를 완벽하게 모듈화하는 데 필수적으로 사용됩니다.
