# 01_핵심_비즈니스_로직과_부가_기능의_혼재

## 상위 맥락
01_기존_코드의_문제점 < 01_AOP_기본_컨셉 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
기존 애플리케이션의 서비스 코드를 살펴보면, 핵심 비즈니스 로직과 부가 기능(횡단 관심사)이 하나의 메서드 내에 심하게 혼재되어 있다는 근본적인 문제점이 있습니다.

핵심 비즈니스 로직과 부가 기능 혼재의 구체적인 양상 메서드는 크게 본연의 목적인 **핵심 관심사항(비즈니스 로직)**과 이를 보조하는 **횡단 관심 사항(부가 로직)**으로 구성됩니다.
예를 들어 데이터베이스에 데이터를 저장하는 insert() 메서드를 작성할 때, 순수하게 비즈니스 데이터를 저장하는 로직(repo.insert)은 단 한 줄에 불과할 수 있습니다.
하지만 이 코드가 정상적으로 작동하기 위해 다음과 같은 다양한 부가 기능 코드가 앞뒤로 추가됩니다.

메서드 실행 상태를 기록하는 로깅
시작 및 종료 시간을 계산하여 소요 시간을 구하는 성능 측정
DB 커넥션을 획득하고 커밋 또는 롤백을 수행하는 트랜잭션 처리
보안, 암호화, 그리고 일괄적인 예외 처리 등
이러한 혼재가 기존 코드에 미치는 문제점 비즈니스 로직과 부가 기능이 뒤섞이면서 다음과 같은 여러 가지 문제들이 발생합니다.
반복적인 코드 중복 (파편화): 로깅, 성능 측정, 트랜잭션 처리와 같은 횡단 관심사는 insert()뿐만 아니라 update(), delete() 등 시스템 내의 여러 메서드를 가로지르며 공통적으로 등장합니다.
이 때문에 알고 보면 모두 비슷한 코드임에도 불구하고 매번 복사하고 붙여넣는 작업이 반복됩니다.

가독성 저하와 주객전도: 핵심이 되어야 할 비즈니스 로직이 각종 연결 설정, 예외 처리, 로그 출력 등의 인프라성 코드에 파묻히게 되어 코드가 본래 의도한 바를 한눈에 파악하기 어렵게 만듭니다.

유지보수 난이도 증가: 부가 기능 코드들이 흩어져 있기 때문에, 특정 부가 기능(예: 보안 방식이나 로깅 정책)을 수정해야 할 경우 연관된 모든 메서드를 찾아가 일일이 고쳐야 합니다.
또한 개발자가 세부적으로 알아야 할 기술들이 증가하여 유지보수가 어려워집니다.

시스템 신뢰성 저하: 다루어야 할 기능이 섞여서 복잡해지면, 특히 초급 개발자의 경우 해당 메서드 내에서 모든 부가 로직을 누락이나 실수 없이 정상적으로 처리했는지 스스로 확신하기 어려워집니다.

해결 방안으로서의 AOP 이러한 기존 코드의 한계를 극복하기 위해 AOP(관점 지향 프로그래밍) 개념이 도입되었습니다.
AOP는 코드 전체에 흩어진 횡단 관심사(보안, 트랜잭션, 로깅 등)를 개별적인 관점으로 바라보고 별도의 모듈로 분리합니다.
이를 통해 중복 코드를 재사용 가능한 형태로 관리할 수 있으며, 개발자는 복잡한 부가 기능 구현에서 벗어나 오직 핵심 비즈니스 로직 구현에만 전념할 수 있게 됩니다.
