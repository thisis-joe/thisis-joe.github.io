# 03_개발자가_핵심_비즈니스_로직에_전념하도록_지원

## 상위 맥락
03_AOP의_정의 < 01_AOP_기본_컨셉 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍)의 정의는 애플리케이션의 전체 기능을 단순한 흐름으로 파악하지 않고, '필요한 개별 관점(Aspect)'으로 분리하여 바라보고 모듈화하는 프로그래밍 패러다임입니다.
이러한 AOP의 정의적 특징이 지향하는 궁극적인 목적이자 가장 핵심적인 효과가 바로 **'개발자가 핵심 비즈니스 로직에만 전념하도록 지원하는 것'**입니다.
AOP의 정의와 기본 컨셉이 개발자의 비즈니스 로직 집중을 돕는 흐름은 다음과 같습니다.
관심사의 엄격한 분리를 통한 인지적 부담 감소: 기존의 개발 방식에서는 하나의 서비스 메서드(예: insert, delete 등) 내부에 데이터를 저장하는 **핵심 관심사(비즈니스 로직)**뿐만 아니라 로깅, 성능 측정, 트랜잭션 관리와 같은 **횡단 관심사(부가 기능)**가 복잡하게 얽혀 있었습니다.
이로 인해 개발자는 비즈니스 로직 외에도 세부적으로 알아야 할 부수적인 기술들이 증가하여 학습 부담이 커지고 정상 로직 처리 여부를 확신하기 어려웠습니다.
AOP는 이 두 가지를 별도의 관점으로 철저히 분리함으로써, 개발자가 부가 기능 구현이라는 불필요한 인지적 부담에서 벗어나게 해줍니다.

횡단 관심사의 모듈화와 런타임 자동 결합: AOP는 시스템 전반에 중복으로 등장하는 보안, 트랜잭션, 예외 처리 등의 부가 기능들을 모아 애스펙트(Aspect)라는 단위로 독립시켜 모듈화합니다.
이렇게 떼어낸 부가 기능(Advice)은 개발자가 비즈니스 코드에 일일이 직접 작성하지 않아도, 스프링의 **프록시(Proxy) 메커니즘을 통해 실행 시점(Runtime)에 타겟(Target) 객체의 앞뒤로 자동 결합(Weaving)**됩니다,.
개발자는 단지 부가 기능이 적용될 위치(Pointcut)만 설정해 두면 됩니다,.

재사용성 극대화 및 비즈니스 로직의 순수성 유지: 분리된 횡단 관심사를 재사용하게 되면, 서비스 클래스 내부에는 오직 데이터베이스에 자료를 저장하거나 가공하는 등의 '순수한 비즈니스 로직'만 텍스트로 남게 됩니다.
코드가 인프라성 로직에 파묻히지 않으므로 가독성이 비약적으로 상승하며, 향후 로깅이나 트랜잭션 정책이 변경되더라도 비즈니스 코드는 단 한 줄도 수정할 필요가 없게 됩니다.

결론적으로 기능을 모듈화된 관점으로 나누어 바라본다는 AOP의 핵심 정의는 **"횡단 관심의 모듈화 → 재사용 → 개발자는 비즈니스 로직에 전념!!"**이라는 논리적 인과관계로 귀결됩니다.
시스템 인프라와 관련된 공통 부가 로직은 프록시 객체에게 위임하고, 실제 비즈니스를 구현하는 개발자는 온전히 자신의 핵심 목표 달성에만 집중할 수 있도록 아키텍처 수준에서 지원하는 것이 AOP의 진정한 가치입니다,.
