# 01_기능을_관점으로_분리

## 상위 맥락
03_AOP의_정의 < 01_AOP_기본_컨셉 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
**AOP(Aspect Oriented Programming, 관점 지향 프로그래밍)의 핵심 정의는 "기능을 단순한 로직의 나열이 아니라, 필요한 개별 '관점(Aspect)'으로 바라보는 것"**입니다.
기능을 관점으로 분리한다는 것은 코드 내에 혼재되어 있는 목적이 다른 로직들을 각자의 역할에 맞게 독립적인 모듈로 쪼개는 것을 의미하며, 구체적인 원리와 맥락은 다음과 같습니다.
### 1. 핵심 관심사항과 횡단 관심 사항의 분리 전통적인 서비스 코드를 살펴보면, 하나의 메서드(예: DB 삽입 로직) 안에 순수한 '비즈니스 로직(핵심 관심사항)'뿐만 아니라 DB 연결, 트랜잭션 관리, 로깅, 성능 측정과 같은 '부가 기능(횡단 관심 사항)'들이 함께 섞여 있습니다.
여기서 **횡단 관심 사항(Cross-cutting Concerns)**이란 여러 메서드와 시스템 전반을 가로지르며 공통적으로 등장하는 기능들을 말합니다.
이러한 부가 기능들이 비즈니스 로직과 얽혀 있으면 세부적으로 알아야 할 기술이 늘어나고 코드의 유지보수가 매우 어려워집니다.
기능을 관점으로 분리한다는 것은 바로 이 공통된 '횡단 관심 사항'들을 독립적인 관점(Aspect)으로 추출해내는 작업을 말합니다.

### 2. 모듈화와 강한 결합의 해소 객체 지향 방식만으로는 해결하기 어려운 '객체 간의 강한 결합' 문제를 해결하는 것이 AOP의 큰 맥락 중 하나입니다. 기존의 방식에서는 특정 객체(예: 관리자)가 다른 객체(예: 청소부)의 기능을 직접 호출해야만 작동하여 결합도가 높았지만, 관점을 분리하게 되면 객체 간의 결합 없이도 필요한 시점에 부가 기능을 개입시킬 수 있습니다.
AOP에서는 이렇게 분리된 횡단 관심사들을 추상적인 명칭인 **애스펙트(Aspect)**로 정의하고, 실제 동작할 부가 로직은 **어드바이스(Advice)**로 모듈화합니다.

### 3. 개발자의 비즈니스 로직 전념 AOP 정의의 가장 큰 목적이자 결론은 '효율성'과 '관심사의 분리'입니다. 분리된 관점(Aspect)은 언제든지 재사용이 가능해집니다.
결론적으로 부가 기능들을 관점으로서 철저히 분리해 내면, 개발자는 반복적이고 복잡한 인프라 로직(트랜잭션, 로깅 등)에 신경 쓰지 않고 오로지 핵심 '비즈니스 로직'의 구현에만 전념할 수 있게 됩니다.

요약하자면, AOP의 큰 틀에서 "기능을 관점으로 분리한다"는 것은 코드를 '비즈니스 핵심 목적'과 '공통 부가 목적'이라는 두 가지 시선으로 나누어 바라보고, 중복되는 부가 목적들을 하나의 독립된 모듈(Aspect)로 떼어내어 코드의 응집도를 높이고 복잡성을 낮추는 핵심 패러다임을 의미합니다.
