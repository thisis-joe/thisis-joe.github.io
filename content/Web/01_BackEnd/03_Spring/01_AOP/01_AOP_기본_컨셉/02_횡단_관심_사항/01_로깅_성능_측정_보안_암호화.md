# 01_로깅_성능_측정_보안_암호화

## 상위 맥락
02_횡단_관심_사항 < 01_AOP_기본_컨셉 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
**횡단 관심 사항(Cross-cutting Concerns)**이란 애플리케이션의 핵심 비즈니스 로직은 아니지만, 여러 메서드와 모듈에 걸쳐 공통적이고 반복적으로 등장하는 부가 로직들을 의미합니다.
로깅, 성능 측정, 보안, 암호화는 모두 이러한 횡단 관심 사항의 대표적인 예시이며, 관점 지향 프로그래밍(AOP)을 통해 핵심 로직과 분리되어 관리됩니다.

각 요소들이 횡단 관심 사항이라는 큰 틀 안에서 어떻게 설명되고 있는지 구체적으로 살펴보면 다음과 같습니다.
### 1. 로깅(Logging)과 성능 측정(Performance Measurement)
문제점: 기존의 서비스 코드에서는 비즈니스 로직(예: DB에 자료 저장) 앞뒤로 현재 시간을 측정하거나 로그를 남기는 코드가 혼재되어 있었습니다.
이러한 코드가 모든 메서드마다 반복되면 코드가 복잡해지고 유지보수가 매우 어려워집니다.

AOP 적용 관점: AOP는 로깅과 성능 모니터링을 독립적인 '관점(Aspect)'으로 분리합니다.
예를 들어, 특정 패키지 내의 DAO 메서드가 호출될 때 전달되는 파라미터를 출력하는 로깅 작업은 핵심 로직을 수정할 필요 없이 별도의 모듈에서 일괄적으로 처리할 수 있습니다.

### 2. 보안 (Security / 인증 및 권한 검사)
적용 관점: 보안 검사 또한 시스템 전반의 여러 메서드(methodA, methodB 등)를 실행하기 전에 공통적으로 수행되어야 하는 횡단 관심사입니다.
시스템은 사용자의 요청이 비즈니스 로직에 도달하기 전에 '인증 및 권한 검사 관점'이라는 별도의 레이어를 거치게 하여, 개발자가 각 기능마다 보안 코드를 중복해서 작성하는 것을 방지합니다.

### 3. 암호화 (Encryption) 및 데이터 마스킹
파라미터 조작을 통한 암호화: 보안 강화를 위한 비밀번호 암호화 처리는 대상 메서드가 실행되기 전에 동작하는 @Before Advice를 활용하여 구현할 수 있습니다.
이를 통해 사용자 가입 등의 핵심 비즈니스 로직으로 데이터가 넘어가기 전, 전달받은 파라미터(비밀번호 평문)를 가로채어 안전하게 암호화한 후 전달하는 방식입니다.

리턴 값 조작을 통한 마스킹: 반대로 데이터를 클라이언트에게 반환할 때는 @AfterReturning Advice를 활용하여 횡단 관심사를 적용할 수 있습니다.
핵심 메서드가 정상적으로 반환한 객체 정보를 가로채어, 비밀번호와 같은 민감한 정보의 리턴 값을 별표(*) 등으로 마스킹 조작한 뒤 반환함으로써 데이터 보안을 한층 두텁게 만듭니다.

정리: 결과적으로 로깅, 성능 측정, 보안, 암호화와 같은 부가 기술들은 세부적으로 알아야 할 내용이 많아 초급 개발자가 모든 비즈니스 로직에 완벽히 적용하기엔 한계가 있습니다.
따라서 AOP를 통해 이러한 횡단 관심사들을 모듈화하여 재사용함으로써, 코드의 결합도를 낮추고 개발자는 오직 핵심 비즈니스 로직을 구현하는 데에만 전념할 수 있게 해주는 것이 가장 핵심적인 연관성입니다.
