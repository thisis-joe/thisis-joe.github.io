# 02_예외_처리_트랜잭션_관리

## 상위 맥락
02_횡단_관심_사항 < 01_AOP_기본_컨셉 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
**예외 처리(Exception Handling)**와 **트랜잭션 관리(Transaction Management)**는 소프트웨어 개발 시 여러 모듈에 걸쳐 공통적으로 등장하는 **횡단 관심 사항(Cross-cutting Concerns)**의 대표적인 사례입니다.

AOP(관점 지향 프로그래밍)의 맥락에서 이 두 가지 요소가 어떻게 다루어지는지 논리적 흐름에 따라 설명해 드리겠습니다.
### 1. 횡단 관심 사항으로서의 예외 처리와 트랜잭션 관리
일반적인 애플리케이션의 메서드는 **'핵심 관심 사항(비즈니스 로직)'**과 이를 보조하는 **'횡단 관심 사항(부가 로직)'**으로 구성됩니다.
과거의 전통적인 서비스 코드를 살펴보면, 순수한 데이터 처리 로직(예: repo.insert()) 앞뒤로 DB 커넥션 연결, 트랜잭션 시작(con.setAutoCommit(false)), 정상 처리 시 커밋(con.commit()), 예외 발생 시 롤백(con.rollback())을 수행하는 코드가 복잡하게 얽혀 있습니다.

이러한 트랜잭션 처리와 예외 처리는 insert, delete, update 등 거의 모든 비즈니스 메서드를 가로지르며 반복적으로 등장합니다.
이처럼 로직이 혼재되어 있으면 코드의 가독성이 떨어질 뿐만 아니라, 세부적인 기술 요소가 증가함에 따라 유지 보수가 매우 어려워지는 문제가 발생합니다.

### 2. AOP를 통한 트랜잭션 관리의 모듈화
AOP는 이러한 문제를 해결하기 위해 기능을 개별 **관점(Aspect)**으로 모듈화합니다.
트랜잭션 처리를 횡단 관심 사항으로 분리하게 되면, 개발자는 트랜잭션의 시작, 커밋, 롤백과 같은 인프라적 코드를 비즈니스 로직 코드 내부에 직접 작성할 필요가 없습니다.
대신, 공통된 트랜잭션 처리 로직을 하나의 모듈로 묶어 두고, 이를 필요로 하는 타겟 메서드에 런타임 프록시(Proxy) 형태로 덧붙여(Weaving) 동작하게 만듭니다.
결과적으로 개발자는 비즈니스 로직 구현에만 온전히 전념할 수 있게 되며, 코드의 재사용성 역시 크게 향상됩니다.

### 3. 일괄적인 예외 처리를 위한 메커니즘
횡단 관심 사항의 또 다른 주요 특징으로 **'일괄적인 예외 처리'**를 꼽습니다.
AOP 환경에서는 비즈니스 로직 내부에 일일이 try-catch 문을 작성하지 않고도 발생한 예외를 우아하게 제어할 수 있습니다.

스프링 AOP는 이를 위해 **@AfterThrowing**이라는 특정 Advice 타입을 제공합니다.

동작 원리: 타겟 메서드(비즈니스 로직)가 실행되던 중 예외를 던지는(throw) 경우에만 이 Advice가 가로채어 동작합니다.

활용: 예외가 발생했을 때 단순히 실행만 되는 것이 아니라, 타겟 메서드에서 던져진 예외 객체를 직접 전달받아 어떤 예외인지 확인하고 추가적인 조작을 가할 수 있습니다.
예를 들어, 특정 예외가 발생했을 때 즉시 시스템 관리자에게 알림 메일을 전송하거나, 오류 로그를 통일된 양식으로 남기는 등의 예외 대책 수립을 전역적으로 일관성 있게 처리할 수 있습니다.

요약
결론적으로 예외 처리와 트랜잭션 관리는 단일 비즈니스 로직에 국한되지 않고 시스템 전반에 걸쳐 요구되는 핵심적인 횡단 관심 사항입니다.
AOP는 이들을 비즈니스 로직과 강하게 결합하는 대신, 독립적인 모듈(Aspect)로 분리하여 코드의 응집도를 높이고 복잡성을 낮추는 구조적 해결책을 제공합니다.
