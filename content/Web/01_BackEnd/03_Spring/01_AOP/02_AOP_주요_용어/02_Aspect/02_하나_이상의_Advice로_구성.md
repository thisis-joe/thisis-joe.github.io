# 02_하나_이상의_Advice로_구성

## 상위 맥락
02_Aspect < 02_AOP_주요_용어 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
**애스펙트(Aspect)**는 AOP 환경에서 로깅, 성능 측정, 트랜잭션 등 시스템 곳곳에 흩어진 모듈화된 횡단 관심사(관점)들을 지칭하는 추상적인 명칭입니다.
이러한 Aspect는 구조적으로 **"하나 이상의 Advice로 구성"**된다고 명시되어 있습니다.

이러한 구성 원리는 Aspect의 본질적인 목적을 잘 보여주는 특징이며, 구체적인 연관성과 인과관계는 다음과 같습니다.
### 1. 추상적 관점(Aspect)과 구체적 행위(Advice)의 관계 Aspect가 '어떤 부가 기능(예: 트랜잭션 관점, 보안 관점)'을 모듈화할 것인지를 나타내는 포괄적인 묶음(컨테이너)이라면, **Advice는 그 관점을 실현하기 위해 실제로 동작하는 모듈화된 '메서드(코드)'**를 의미합니다.
하나의 추상적인 관점이 실제 시스템에서 동작하기 위해서는, 타겟(Target) 객체의 실행 흐름 속에서 '어떤 상황(@Before, @After 등)'에 '무엇을 할 것인지'를 정의한 실체적인 행위(Advice)가 반드시 하나 이상 포함되어야만 합니다.

### 2. 복합적인 실행 시점 통제 (왜 '하나 이상'인가?)
단일 Advice 구성: 단순한 로깅 관점(Logging Aspect)의 경우, 타겟 메서드 실행 직전에만 파라미터나 호출 정보를 로그로 남기는 단 하나의 @Before Advice만으로도 충분히 구성될 수 있습니다.

다중 Advice 구성: 그러나 시스템에서 요구하는 관점은 단편적이지 않은 경우가 많습니다.
대표적으로 **'트랜잭션 관리 관점'**을 들 수 있습니다.
하나의 완전한 트랜잭션 관점을 시스템에 적용하려면 다음과 같은 여러 시점의 조작이 동시에 필요합니다.
비즈니스 로직 실행 전: 데이터베이스 커넥션을 얻고 트랜잭션을 시작 (@Before 동작에 해당)
비즈니스 로직 정상 종료 후: 변경 사항을 커밋 (@AfterReturning 동작에 해당)
비즈니스 로직 중 예외 발생 시: 변경 사항을 롤백 (@AfterThrowing 동작에 해당)
만약 이 세 가지 동작이 개별적으로 흩어져 있다면 관리가 불가능할 것입니다.
따라서 Aspect는 이처럼 하나의 공통된 목적(트랜잭션)을 달성하기 위해 다양한 시점(Join point)에서 동작해야 하는 '여러 개의 Advice'들을 하나의 논리적인 단위로 묶어내는 역할을 합니다.

### 3. 실제 코드 레벨에서의 구조화 이론적인 "하나 이상의 Advice로 구성된다"는 특징은 실제 스프링 프로그래밍 환경에서 클래스와 메서드의 관계로 나타납니다. 개발자가 횡단 관심사를 작성할 때, 클래스 단위에 @Aspect 어노테이션을 선언하여 하나의 '관점'을 정의합니다.
그리고 그 클래스 내부에는 횡단 관심 코드를 담고 있는 **여러 개의 메서드(Advice)**들을 선언하게 되며, 각 메서드에는 @Before, @After, @Around 등의 어노테이션을 붙여 각자의 실행 시점을 지정합니다.

결론적으로, 기능을 개별 관점으로 분리하여 관리한다는 AOP의 큰 틀 속에서 Aspect가 "하나 이상의 Advice로 구성된다"는 것은, 복잡하고 다각적으로 이루어지는 부가 기능(Advice)들을 하나의 응집도 높은 모듈(Aspect)로 캡슐화하여 개발자가 쉽게 재사용하고 관리할 수 있도록 설계된 핵심적인 구조를 의미합니다.
