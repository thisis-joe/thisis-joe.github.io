# 01_Advice_적용_가능_지점

## 상위 맥락
04_Join_point < 02_AOP_주요_용어 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 용어 체계에서 **조인 포인트(Join point)**는 횡단 관심사 모듈인 어드바이스(Advice)가 개입하여 적용될 수 있는 모든 실행 가능한 지점을 의미합니다.

이론적인 AOP 패러다임과 실제 스프링(Spring) 환경에서의 구체적인 구현을 나누어 'Advice 적용 가능 지점'의 맥락을 살펴보면 다음과 같습니다.
### 1. 이론적 관점에서의 다양한 Advice 적용 가능 지점 원론적인 AOP 설계 개념에 따르면, 횡단 관심사(부가 기능)가 개입할 수 있는 조인 포인트는 프로그램의 실행 흐름 전체에 걸쳐 매우 광범위하게 열려 있습니다. 구체적으로는 객체의 필드 값이 변경되는 순간, 특정 메서드가 실행되는 순간, 객체의 생성자가 호출되는 순간 등 여러 지점들이 모두 Advice를 적용할 수 있는 조인 포인트의 후보가 될 수 있습니다.

### 2. 스프링(Spring) AOP의 한계와 제한: '메서드 실행 시점' 하지만 스프링 환경에서는 Advice를 적용할 수 있는 조인 포인트가 철저하게 '메서드 실행 시점' 하나로만 엄격하게 제한됩니다.
그 이유는 스프링 AOP가 타겟(Target) 객체의 바이트코드를 직접 조작하는 대신, 프록시(Proxy) 빈(Bean)을 기반으로 동작하기 때문입니다.
프록시 객체는 클라이언트의 요청을 대리하여 가로채는 방식으로 동작하므로, 인터페이스나 클래스의 '메서드 호출' 단계에는 개입할 수 있지만, 객체 내부의 순수한 필드 값 변경이나 생성자 초기화 과정에는 개입할 수 없다는 구조적 특징을 가집니다.

### 3. 조인 포인트를 기준으로 세분화되는 Advice 타입 유일한 Advice 적용 가능 지점이 '타겟 메서드의 실행 시점'으로 한정됨에 따라, 실제 부가 기능을 수행하는 Advice들은 이 메서드 실행 흐름의 어느 타이밍에 동작할 것인지에 따라 5가지 타입으로 구체적으로 나뉘어 규정됩니다.

@Before: 타겟 메서드가 호출되기 전 (파라미터 조작 가능).

@AfterReturning: 타겟 메서드가 정상적으로 종료(return)된 경우 (리턴 값 조작 가능).

@AfterThrowing: 타겟 메서드 실행 중 예외가 던져졌을 때.

@After: 정상 처리 및 예외 발생 여부와 무관하게 언제나 실행 (마치 finally 블록처럼 동작).

@Around: 타겟 메서드 실행 전후의 모든 과정을 둘러싸고 완벽하게 제어.

정리하면 Join point라는 더 큰 맥락에서 **'Advice 적용 가능 지점'**은 횡단 관심사가 개입할 수 있는 이론적 기회들을 총망라하는 개념입니다.
그러나 실무적인 스프링 AOP 아키텍처 내에서는 프록시 메커니즘이라는 기술적 기반에 의해 오직 '타겟 객체의 메서드 실행 시점'으로 한정되며, 이 단일화된 조인 포인트를 기준으로 다섯 가지 형태의 Advice들이 타겟의 앞, 뒤, 혹은 주변에 유기적으로 결합(Weaving)되어 AOP의 관심사 분리를 완성하게 됩니다.
