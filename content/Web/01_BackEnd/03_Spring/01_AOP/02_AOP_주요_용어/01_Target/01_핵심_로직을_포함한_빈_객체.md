# 01_핵심_로직을_포함한_빈_객체

## 상위 맥락
01_Target < 02_AOP_주요_용어 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍)의 핵심 용어 중 하나인 Target(타겟)은 명확하게 "핵심 비즈니스 로직을 포함한 빈(Bean) 객체"를 의미합니다.

AOP의 기본 컨셉은 공통적으로 등장하는 부가 기능(횡단 관심사)을 핵심 로직에서 떼어내어 모듈화하는 것입니다.
이 맥락에서 Target과 그것이 품고 있는 '핵심 로직을 포함한 빈 객체'가 가지는 의미와 역할은 다음과 같습니다.
### 1. AOP가 적용될 최종 목적지 (순수 비즈니스 로직의 결정체) 기존의 코드에서는 데이터 저장과 같은 핵심 로직 주변에 로깅, 트랜잭션, 보안 등의 부가 기능이 혼재되어 있었습니다.
하지만 AOP를 적용하게 되면, 이러한 부가 기능들은 모두 Aspect(애스펙트)로 분리되어 떨어져 나갑니다.
결국 모든 부가 기능이 제거되고 오직 시스템 본연의 "순수한 비즈니스 목적"만을 수행하는 주체로 남게 되는 객체가 바로 Target입니다.

### 2. 프록시(Proxy) 메커니즘을 통한 보호와 위임 Spring AOP 환경에서 클라이언트는 Target 객체를 직접 호출하지 않습니다. 대신 Target을 감싸고 있는 **프록시(Proxy)**라는 대리자 빈 객체가 사용됩니다.

동작 원리: 프록시 객체가 클라이언트의 호출을 가로채어 자신에게 연결된 부가 기능(Advice)을 먼저 실행합니다.
그 후, 실제 핵심 로직을 쥐고 있는 Target 객체의 메서드를 내부적으로 호출(위임)하여 비즈니스 로직을 마저 수행하도록 합니다.

결과적으로 Target 객체 내부의 코드는 전혀 수정되지 않은 상태에서, 외부의 프록시를 통해 런타임에 동적으로 부가 기능이 결합(Weaving)되는 구조를 갖습니다.

### 3. 모든 부가 기능(Advice) 실행의 절대적 기준점 AOP에서 부가 기능을 구현한 메서드인 Advice는 반드시 Target 객체를 기준으로 언제 실행될지가 결정됩니다.
Target(핵심 로직을 포함한 빈 객체) 내부의 메서드 실행 시점(Join point)에 따라 Advice의 타입이 엄격하게 규정됩니다.

@Before: Target의 메서드가 호출되기 직전에 부가 로직을 실행합니다.

@AfterReturning: Target의 메서드가 정상적으로 완료되어 결과값을 반환(return)한 경우에만 개입하여 동작합니다.

@AfterThrowing: Target의 메서드 내부 비즈니스 로직을 수행하던 중 예외가 발생해 던져졌을(throw) 때 동작합니다.

@After: 예외 발생 여부와 상관없이 Target 메서드의 실행이 종료되면 무조건 실행됩니다.

@Around: Target의 메서드 실행 전과 후를 모두 둘러싸며 제어권을 가집니다.
내부에서 proceed()를 호출하여 실제 Target의 핵심 로직을 실행시킬 수 있으며, 그 과정에서 Target에 전달될 파라미터나 Target이 뱉어낸 리턴 값을 완전히 조작하거나 대체하는 것도 가능합니다.

요약하자면, **Target(핵심 로직을 포함한 빈 객체)은 AOP가 보호하고 분리하고자 하는 애플리케이션의 '가장 중요한 코어'**입니다.
부가 기능의 구현과 실행 타이밍 책임을 모두 외부의 프록시와 Advice에 위임함으로써, Target은 복잡한 인프라 로직의 개입 없이 오직 비즈니스 로직에만 고도로 집중할 수 있는 완벽한 객체 지향적 모듈화를 달성하게 됩니다.
