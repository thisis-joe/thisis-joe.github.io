# 01_Join_point_선별을_위한_표현식

## 상위 맥락
05_Pointcut < 02_AOP_주요_용어 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍)의 용어 체계 속에서 Pointcut(포인트컷)은 "어드바이스(Advice)를 적용할 대상을 결정하는 표현식", 즉 수많은 Join point 중에서 실제로 부가 기능을 동작시킬 지점을 정밀하게 선별해 내는 필터로 정의됩니다.

이러한 Join point 선별을 위한 표현식이 Pointcut이라는 더 큰 개념 속에서 어떻게 구성되고 동작하는지 논리적 흐름에 따라 구체적으로 살펴보면 다음과 같습니다.
### 1. Join point와 Pointcut의 상관관계 (필터링 역할) Spring AOP 환경에서 Join point는 부가 기능(Advice)이 개입할 수 있는 후보 지점들로, 내부적으로 프록시 방식을 사용하기 때문에 오직 **'메서드 실행 시점'**으로만 제한됩니다.
하지만 시스템 상의 모든 메서드에 부가 기능을 적용할 수는 없으므로, Pointcut이라는 정교한 '표현식'을 필터로 사용하여 무수히 많은 Join point(메서드 실행 시점) 중 특정 조건에 부합하는 타겟(Target)의 메서드만을 선별해 냅니다.

### 2. 선별을 위한 다양한 지정자 (Designator) 표현식을 작성할 때는 어떤 기준으로 메서드를 선별할 것인지 결정하기 위해 다양한 지정자를 활용합니다.

execution: 메서드의 시그니처(리턴 타입, 패키지, 클래스, 메서드 이름, 파라미터)를 직접 명시하여 가장 정교하게 타겟을 선별할 수 있는 표현식입니다.

within: 특정 클래스나 패키지 내부에 속해 있는 모든 메서드를 일괄적으로 선별합니다.

bean: 스프링 컨테이너에 등록된 빈(Bean)의 구체적인 이름을 기반으로 선별합니다.

@annotation / @within: 특정 애너테이션이 선언된 메서드나, 해당 애너테이션이 적용된 빈의 모든 메서드를 기준으로 선별합니다.

### 3. 와일드카드를 통한 유연한 표현식 작성 가장 널리 사용되는 execution 지정자를 사용할 때, 개발자가 수많은 대상 메서드를 일일이 타이핑하는 것은 비효율적이므로 와일드카드를 활용하여 범용적인 선별 표현식을 작성할 수 있습니다.

`*` (별표): 타입에 무관함을 나타내거나, 메서드 이름 등의 임의의 문자열 대체를 위해 사용하며, 파라미터 위치에서는 정확히 '1개의 파라미터'를 의미합니다.
`..` (점 두 개): 패키지 경로에서 0개 이상의 하위 패키지를 한 번에 대체하거나, 파라미터 부분에 작성하여 파라미터의 타입과 개수에 무관하게 '0개 이상의 파라미터'를 허용하도록 만듭니다.

### 4. 표현식의 결합 및 재사용 (Named Pointcut) Pointcut 표현식은 단순히 하나의 조건으로 끝나지 않고, 논리 연산자(&&, ||, !, and, or, not)를 사용하여 여러 개의 조건을 복합적으로 연결함으로써 더욱 세밀한 선별망을 구축할 수 있습니다.
또한, 매번 Advice에 길고 복잡한 표현식을 중복해서 적는 대신, @Pointcut 애너테이션을 빈 메서드에 선언하고 그 메서드의 이름을 호출하는 방식(Named pointcut)으로 한 번 작성한 선별 표현식을 여러 Advice에서 손쉽게 재사용할 수도 있습니다.

정리하면 "Join point 선별을 위한 표현식"은 Pointcut의 본질 그 자체를 나타냅니다.
AOP 아키텍처가 횡단 관심사(Aspect)를 핵심 비즈니스 로직(Target)의 코드 수정 없이 동적으로 엮어낼(Weaving) 수 있는 이유는, 이 Pointcut 표현식이 "언제, 어디서 부가 기능을 실행해야 하는가?"에 대한 정확한 좌표계를 시스템에 제공하기 때문입니다.
이를 통해 개발자는 고도로 모듈화된 부가 기능들을 필요한 곳에만 정밀하게 타겟팅하여 적용할 수 있습니다.
