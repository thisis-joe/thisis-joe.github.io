# 01_Aspect를_비즈니스_로직에_적용하는_행위

## 상위 맥락
06_Weaving < 02_AOP_주요_용어 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍)의 주요 용어 체계에서 "Aspect(애스펙트)를 비즈니스 로직 코드에 적용하는 행위 자체"를 바로 위빙(Weaving)이라고 정의합니다.

위빙은 AOP의 근본 목적인 '관심사의 분리'를 실제 시스템에서 동작 가능하게 만들어주는 핵심적이고 최종적인 결합 메커니즘입니다.
위빙이 어떠한 논리적 흐름과 인과관계를 거쳐 비즈니스 로직에 Aspect를 적용하는지 세부적으로 살펴보면 다음과 같습니다.
### 1. 결합의 대상 (무엇을 엮는가?) 위빙은 크게 두 가지 요소의 결합을 의미합니다. 하나는 공통 부가 기능(횡단 관심사)을 모듈화해 둔 추상적 명칭인 **Aspect(애스펙트)**이며, 다른 하나는 순수한 핵심 비즈니스 로직을 포함하고 있는 빈 객체인 **Target(타겟)**입니다.
기존 방식에서는 이 두 가지가 하나의 메서드 내에 섞여 있었지만, AOP에서는 물리적으로 완전히 분리된 이 두 모듈을 실행 시점에 하나로 엮어내는 행위가 필요하며 이것이 바로 위빙입니다.

### 2. 스프링 AOP의 위빙 메커니즘 (프록시 기반) 스프링 환경에서 이 적용 행위(Weaving)는 원본 Target 객체의 바이트코드를 뜯어고치는 형태가 아니라, 프록시(Proxy)라는 가짜 객체를 활용하여 런타임(실행 시점)에 간접적으로 엮어내는 방식을 사용합니다.

스프링 컨테이너는 빈을 생성할 때, 포인트컷(Pointcut)을 검토하여 해당 Target이 부가 기능(Aspect) 적용 대상인지 판단합니다.

적용 대상이라면, 컨테이너는 순수한 Target을 감싸는 대리인 객체인 프록시(Proxy)를 생성하여 클라이언트에게 반환합니다.

결과적으로 클라이언트는 Target을 직접 호출하는 것이 아니라 프록시를 호출하게 되며, 이 프록시 내부에서 위빙된 흐름이 동작합니다.

### 3. 위빙의 구체적인 실행 흐름 (Advice의 개입) 프록시에 의해 위빙이 성립되면, 프록시는 클라이언트의 요청을 가로챈 후 지정된 **조인 포인트(Join point, 메서드 실행 시점)**를 기준으로 Aspect 내부의 구체적인 실행 코드인 **Advice(어드바이스)**들을 비즈니스 로직 전후에 배치하여 실행시킵니다.

타겟의 비즈니스 로직 실행 전에는 파라미터를 조작하는 @Before Advice가 동작하도록 엮어냅니다.

내부적으로 타겟의 실제 비즈니스 로직 메서드를 호출합니다.

비즈니스 로직이 정상 종료되면 결과값을 조작하는 @AfterReturning Advice를, 예외가 던져지면 대책을 수립하는 @AfterThrowing Advice를 엮어내어 동작시킵니다.

정리하면 **위빙(Weaving)**은 "분리된 것을 다시 합친다"는 단순한 의미를 넘어, 핵심 비즈니스 로직 코드의 순수성을 완벽하게 보존하면서도 런타임에 동적으로 부가 기능을 결합해 내는 AOP 아키텍처의 꽃입니다.
Aspect를 비즈니스 로직에 적용하는 이 모든 행위(Weaving)를 프록시 객체와 스프링 프레임워크가 배후에서 전담해 주기 때문에, 비즈니스 로직 내부에는 더 이상 인증이나 트랜잭션과 같은 부가 기능 코드가 존재할 필요가 없습니다.
이를 통해 개발자는 공통 인프라 로직에 얽매이지 않고 오직 핵심 비즈니스 로직의 구현에만 전념할 수 있게 되는 AOP의 궁극적인 설계 목적이 달성됩니다.
