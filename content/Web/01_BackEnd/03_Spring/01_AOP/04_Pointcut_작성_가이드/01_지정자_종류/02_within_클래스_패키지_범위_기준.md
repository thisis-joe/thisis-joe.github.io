# 02_within_클래스_패키지_범위_기준

## 상위 맥락
01_지정자_종류 < 04_Pointcut_작성_가이드 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍)의 아키텍처에서 **Pointcut(포인트컷)**은 무수히 많은 조인 포인트(메서드 실행 시점) 중 부가 기능(Advice)을 적용할 대상을 선별해 내는 필터 역할을 합니다.
이때 대상을 필터링하는 '기준'을 무엇으로 삼을 것인가를 결정하는 핵심 키워드들을 **지정자(Designator)**라고 부르며, within은 그중에서도 '클래스 또는 패키지의 물리적 위치 범위'를 필터링 기준으로 삼는 지정자입니다.

다양한 지정자 종류의 전체적인 맥락 속에서 within의 원리와 구조적 역할을 분석하면 다음과 같습니다.
### 1. Pointcut 지정자(Designator)의 종류와 역할 분담 스프링 AOP는 부가 기능이 엮일 타겟(Target)을 정확히 조준하기 위해 다음과 같이 각기 다른 판별 원리를 가진 지정자들을 제공합니다.

execution: 메서드의 리턴 타입, 패키지 및 클래스명, 메서드명, 파라미터 등 시그니처 전체를 조합하여 가장 정밀하게 대상을 선별합니다.

within: 특정 클래스 내부나, 특정 패키지 산하에 위치한 모든 메서드를 범위 단위로 선별합니다.

bean: 스프링 컨테이너에 등록된 빈(Bean)의 고유 이름을 기준으로 선별합니다.

@within / @annotation: 타겟의 빈 객체나 메서드에 선언된 메타데이터(특정 애너테이션)의 유무를 기준으로 선별합니다.

### 2. within 지정자의 동작 원리: 클래스 및 패키지 범위 일괄 지정 within 지정자는 타겟 메서드의 구체적인 이름(method_name)이나 파라미터(parameter), 리턴 타입(return_type)에는 전혀 관여하지 않고, 해당 메서드가 소속된 '물리적인 위치(클래스와 패키지 경로)'만을 평가합니다.

작성 문법의 인과관계: within 지정자의 내부 표현식 작성 방식은, 가장 복잡한 구조를 가진 execution 지정자 표현식 중 'Package + class'를 명시하는 구문 영역과 완벽하게 동일하게 작성됩니다.

적용 예시: @Before("within(com.ssafy..dao)")라고 작성할 경우, 스프링 프레임워크는 com.ssafy 패키지와 그 하위의 모든 패키지(..) 경로에 존재하는 dao라는 명칭과 관련된 객체 내부의 '모든 메서드'를 일괄적으로 AOP 적용 대상으로 판단하여 위빙(Weaving)을 수행합니다.

### 3. 통찰: 다른 주요 지정자들과의 관계 및 차이점
execution과의 상관관계 (정밀 통제 vs 넓은 포괄성): execution이 메서드의 형태를 파라미터 수준까지 아주 정교하게 검사하는 역할을 한다면, within은 특정 아키텍처 계층을 포괄적으로 묶어내는 역할을 합니다.
예를 들어, 파라미터가 1개인 특정 메서드에만 보안 검사를 적용하려면 execution을 써야 하지만, 시스템의 DAO 계층을 구성하는 모든 클래스에 일괄적으로 성능 측정 로그를 남기고자 할 때는 within을 사용하여 패키지 단위로 묶는 것이 구조적으로 훨씬 간결하고 타당합니다.

@within과의 상관관계 (물리적 경로 vs 논리적 메타데이터): 두 지정자 모두 선별된 '클래스 범위 내의 모든 메서드에 적용'된다는 결과적 특징은 동일합니다.
그러나 대상을 판별하는 원인에 있어, within은 소스 코드 파일이 존재하는 **물리적 경로 및 클래스명(문자열)**을 검사하는 반면, @within은 해당 클래스에 @Service와 같은 논리적인 애너테이션이 부착되어 있는지를 검사한다는 명확한 원리적 차이가 있습니다.

결론적으로 지정자 생태계 내에서 **within**은 개발자가 애플리케이션의 물리적 폴더 구조 및 모듈화된 패키지 계층(Controller, Service, Dao 등)을 바탕으로 특정 계층 전체에 횡단 관심사를 쉽고 강력하게 일괄 적용할 수 있도록 지원하는 핵심적인 범위 필터링 기능을 수행합니다.
