# 01_Pointcut을_이용한_재사용

## 상위 맥락
04_named_pointcut < 04_Pointcut_작성_가이드 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP 환경에서 복잡한 포인트컷(Pointcut) 표현식은 여러 어드바이스(Advice)에서 필연적으로 중복 사용될 가능성이 높습니다.
이러한 중복을 제거하고 유지보수성을 극대화하기 위해 제공되는 아키텍처적 기법이 바로 'Named Pointcut(네임드 포인트컷)'이며, 이를 실제 코드로 구현하여 모듈화하는 핵심 도구가 @Pointcut 애너테이션입니다.
@Pointcut을 이용한 재사용 원리와 내부적인 인과관계를 단계별로 분석하면 다음과 같습니다.
### 1. Named Pointcut의 선언과 고유 식별자(ID) 부여 특정 필터링 조건(예: execution(* com.ssafy.service.*.*(..)))을 시스템 전반에서 재사용하기 위해, 개발자는 내부 구현 로직이 완전히 비어있는(// 내용은 필요 없음) void 타입의 일반 메서드를 생성하고 그 상단에 @Pointcut 애너테이션과 함께 표현식을 선언합니다.
이 과정에서 발생하는 가장 핵심적인 구조적 변환은, @Pointcut이 선언된 '메서드의 이름(예: complexExecution)' 자체가 해당 복잡한 포인트컷 표현식을 대표하는 고유한 식별자(ID)로 등록된다는 점입니다.

### 2. 어드바이스(Advice)에서의 호출 방식 및 재사용 이렇게 ID가 부여된 Named Pointcut은 이후 정의되는 수많은 어드바이스(@Before, @After 등)에서 손쉽게 참조하여 사용할 수 있습니다.
어드바이스를 작성할 때 복잡한 표현식을 일일이 하드코딩하는 대신, 애너테이션의 값(value)으로 미리 정의해 둔 포인트컷 메서드의 이름(complexExecution())을 마치 함수를 호출하듯 괄호를 포함한 형태로 작성하여 필터링 조건을 그대로 재사용합니다.

### 3. 외부 클래스 참조 시의 엄격한 경로 명시 규칙 아키텍처를 고도화하다 보면 포인트컷만 모아둔 클래스(예: PointcutAspect)와 실제 부가 기능을 수행하는 클래스(예: LoggingAspect)를 물리적으로 철저히 분리하게 됩니다. 이처럼 Named Pointcut이 외부에 존재하는 경우, 어드바이스에서 이를 참조할 때는 단순히 메서드 이름만 적어서는 안 됩니다. 스프링 컨테이너가 해당 ID를 정확히 찾아낼 수 있도록 "com.ssafy.live.basic.aspect.PointcutAspect.complexExecution()"과 같이 패키지명과 클래스 이름을 모두 포함한 전체 풀 경로(Full path) 형태로 작성해야만 교차 참조 및 완벽한 재사용이 성립됩니다.

정리하면 Named Pointcut이라는 큰 맥락에서 @Pointcut 애너테이션은 "파편화된 타겟팅 조건식들을 하나의 논리적인 모듈(상수)로 캡슐화"하는 강력한 선언적 도구입니다.

이러한 재사용 기법이 적용되면, 훗날 대상 비즈니스 로직(Target)의 시그니처나 패키지 구조가 변경되더라도 시스템 곳곳에 흩어진 어드바이스 코드를 일일이 찾아 수정할 필요가 없습니다.
오직 @Pointcut이 선언된 원본 메서드의 표현식 단 한 곳만 수정하면, 이를 참조(메서드 호출 형태)하고 있는 모든 어드바이스들의 타겟팅 조건이 일괄적으로 동기화되므로 극단적인 수준의 유지보수성과 객체지향적 설계의 이점을 누릴 수 있게 됩니다.
