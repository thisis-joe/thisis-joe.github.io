# 02_package_class_선택

## 상위 맥락
02_execution_표현식_구조 < 04_Pointcut_작성_가이드 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
스프링 AOP(관점 지향 프로그래밍)에서 execution 표현식은 타겟(Target) 메서드의 전체 시그니처를 기반으로 가장 정교하게 조인 포인트(Join point)를 필터링하는 포인트컷(Pointcut) 지정자입니다.
이러한 execution 표현식 구조 내에서 Package + class (선택) 파트가 갖는 역할과 작성 규칙, 그리고 내부적인 동작 메커니즘을 단계별로 상세히 분석하면 다음과 같습니다.
### 1. execution 표현식의 전체 구조와 Package + class의 위치 execution 지정자는 문법적으로 4개의 구역으로 조합됩니다. 가장 앞에 리턴 타입인 return_type(필수)가 오고, 그다음 조인 포인트의 공간적 위치를 지정하는 Package + class(선택), 이어서 실행될 메서드의 이름인 method_name(필수), 마지막으로 파라미터 구조를 나타내는 parameter로 구성됩니다.
이 전체 구조 속에서 Package + class는 메서드가 선언된 구체적인 네임스페이스와 타입 계층을 지정하여 AOP의 개입 범위를 좁히는 필터 역할을 수행합니다.
### 2. Package + class의 선언 규칙 및 인과관계
타입 기반의 한정: 이 항목은 단순히 패키지 경로만 적는 것이 아니라, 패키지를 포함하는 전체 클래스 이름 또는 인터페이스의 타입을 명시하여 대상 메서드의 소속을 결정짓습니다.

선택적 생략과 유연성: 이 항목은 필수 구문이 아닌 '선택사항(Optional)'이므로 생략이 가능합니다.
만약 개발자가 이 항목을 생략한다면, 스프링 AOP는 특정 클래스나 패키지에 국한하지 않고 지정된 method_name과 파라미터 규칙에 부합하는 애플리케이션 내의 모든 메서드를 위빙(Weaving) 대상으로 간주하게 됩니다.
구조적 연결점 (.): 만약 생략하지 않고 패키지와 클래스를 구체적으로 명시할 경우, 이 구역은 반드시 뒤따르는 필수 항목인 메서드 이름(method_name)과 마침표(.)를 통해 논리적으로 연결되어야 합니다.

### 3. 와일드카드를 활용한 동적 패턴 매칭 정적이고 명시적인 하드코딩을 피하고 AOP의 장점인 범용적 적용을 극대화하기 위해, Package + class 구역에는 두 가지 형태의 와일드카드가 결합되어 사용됩니다.
`*` (별표): 0개 이상의 문자열을 대체합니다.
이를 통해 특정 패키지 경로의 이름 일부를 무시하거나, 특정 접미사/접두사를 가진 클래스(예: *Dao, *Service)들만 필터링하는 등 클래스 이름 단위의 패턴 매칭을 수행할 수 있습니다..
`..` (점 두 개): 계층 구조에서 0개 이상의 하위 패키지를 한꺼번에 대체하는 강력한 문법입니다.
예를 들어 com.ssafy..와 같이 작성하면 해당 패키지 밑에 존재하는 모든 뎁스(Depth)의 하위 패키지를 전부 스캔 대상으로 포함시킵니다.
단, 파서(Parser)의 모호성을 방지하기 위해 이 .. 기호는 Package + class 표현식의 맨 처음에는 등장할 수 없다는 엄격한 문법적 제약이 존재합니다.

정리하면 execution 표현식 구조 내에서 Package + class는 타겟 객체가 존재하는 '물리적/논리적 공간의 좌표'를 설정하는 핵심 영역입니다.
이 항목이 '선택사항'으로 설계된 것과 와일드카드(*..)를 완벽하게 지원하는 것은, 개발자가 AOP 적용 대상을 단일 클래스 단위의 미시적인 수준부터 하위 패키지 전체를 아우르는 거시적인 수준까지 자유자재로 조율(Tuning)할 수 있도록 만들기 위함입니다.
이를 통해 핵심 비즈니스 로직(Target)의 코드는 그대로 유지하면서도, 시스템 아키텍처의 패키지 설계 계층에 맞추어 횡단 관심사(Aspect)를 정밀하게 투영시키는 고도화된 모듈화가 실현됩니다.
