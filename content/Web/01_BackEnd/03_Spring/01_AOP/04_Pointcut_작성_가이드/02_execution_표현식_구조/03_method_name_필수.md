# 03_method_name_필수

## 상위 맥락
02_execution_표현식_구조 < 04_Pointcut_작성_가이드 < Aspect_Oriented_Programming_AOP < Spring < Web

## 핵심 정리
AOP(관점 지향 프로그래밍) 체계에서 execution 지정자는 타겟 메서드의 시그니처 전체를 조합하여 가장 정교하게 조인 포인트(Join point)를 필터링하는 포인트컷(Pointcut) 표현식입니다.

이 execution 표현식의 기본 구조는 return_type(필수) [Package + class] method_name(필수) (parameter) 형태로 구성되며, 이 중 **method_name은 대상을 선별하기 위해 단독으로 생략할 수 없는 '필수 입력 사항'**입니다.

execution 표현식 구조 내 method_name의 역할과 인과관계를 심도 있게 살펴보면 다음과 같습니다.
### 1. method_name이 필수 조건이 되는 아키텍처적 인과관계 스프링 AOP(Spring AOP)가 런타임에 프록시(Proxy)를 통해 개입할 수 있는 조인 포인트는 철저하게 '메서드 실행(execution) 시점' 하나로만 엄격히 제한됩니다.
객체의 필드 값 변경이나 생성자 호출 등에는 개입할 수 없기 때문에, 스프링 컨테이너가 런타임에 "어떤 행위를 가로챌 것인가?"를 판단하기 위한 최소한의 식별 기준은 '메서드의 이름'이 될 수밖에 없습니다.
따라서 execution 표현식에서 method_name의 명시는 타겟을 조준하기 위한 필수 불가결한 뼈대입니다.

### 2. 와일드카드(*)를 통한 유연한 패턴 매칭 수많은 비즈니스 로직(Target)의 메서드 이름을 표현식에 일일이 하드코딩하는 것은 비효율적이므로, method_name 작성 시에는 '0개 이상의 문자열'을 대체하는 *(별표) 와일드카드를 적극적으로 결합하여 사용합니다.

예를 들어, 표현식을 set*으로 지정하면 setName, setPassword 등 set으로 시작하는 모든 상태 변경 메서드들을 한 번에 AOP 적용 대상으로 선별할 수 있습니다.

아예 * 단독으로 작성할 경우, 이름에 구애받지 않고 조건에 맞는 클래스 내의 '모든 메서드'를 타겟팅하게 됩니다.

### 3. 전체 시그니처(Signature) 조건과의 유기적 결합 method_name은 필수 요소이지만, 이것 하나만으로는 정교한 필터링이 완성되지 않습니다. 자바(Java)와 같은 객체지향 언어에서는 오버로딩(Overloading)을 통해 동일한 이름의 메서드가 여러 개 존재할 수 있기 때문입니다. 따라서 method_name은 또 다른 필수 입력 사항인 return_type(리턴 타입, 와일드카드 * 사용 가능) 및 괄호 안에 작성되는 parameter(파라미터, 와일드카드 .. 등 사용 가능) 표현식과 반드시 결합되어야 합니다.
예를 들어 execution(* pointcut..*Bean.set*(..))와 같이 반환 타입, 클래스 경로, 메서드 이름, 파라미터 규칙이 하나의 문자열로 결합될 때 비로소 특정한 비즈니스 목적을 가진 정확한 타겟 메서드군을 걸러낼 수 있습니다.

정리하면 execution 표현식 구조에서 method_name(필수)은 프록시가 가로채야 할 행위의 실체를 시스템에 지시하는 최소한의 필수 좌표입니다.
개발자는 핵심 비즈니스 로직을 전혀 수정하지 않고 외부의 @Aspect 클래스에서 이 method_name과 와일드카드(*)를 조합한 표현식을 선언함으로써, 시스템 전반에 걸친 특정 메서드들의 실행 흐름을 일관되고 강력하게 통제(Weaving)할 수 있게 됩니다.
